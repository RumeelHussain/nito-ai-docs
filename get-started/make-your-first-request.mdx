This page takes the single most common call, a non-streaming chat completion, and walks through every part of it. "Non-streaming" means you wait for the whole answer and get it back as one JSON document, which is the simplest shape to learn from. By the end you will understand each field you send and each field you get back, which makes everything else in the docs easier to read. The next page covers [streaming](/get-started/stream-your-first-response), where the same answer arrives piece by piece.

If you are new to chat APIs, the mental model is a transcript. You send a list of messages (the conversation so far, including a system instruction and the user's question), and the model appends one more message: its reply. That is the whole interaction. Everything else on this page is detail around that core.

## The call

<CodeGroup>

```bash curl
curl "https://YOUR-NITO-BASE-URL/v1/chat/completions" \
  -H "Authorization: Bearer $NITO_API_KEY" \
  -H "Content-Type: application/json" \
  -d '{
    "model": "grok-4.3",
    "messages": [
      { "role": "system", "content": "You are a concise assistant." },
      { "role": "user", "content": "Name three benefits of private AI inference." }
    ],
    "temperature": 0.7,
    "max_tokens": 300
  }'
```

```python Python
from openai import OpenAI

client = OpenAI(api_key="z_sk_your_key_here", base_url="https://YOUR-NITO-BASE-URL/v1")

resp = client.chat.completions.create(
    model="grok-4.3",
    messages=[
        {"role": "system", "content": "You are a concise assistant."},
        {"role": "user", "content": "Name three benefits of private AI inference."},
    ],
    temperature=0.7,
    max_tokens=300,
)
print(resp.choices[0].message.content)
print(resp.usage)
```

```typescript TypeScript
import OpenAI from "openai";

const client = new OpenAI({ apiKey: "z_sk_your_key_here", baseURL: "https://YOUR-NITO-BASE-URL/v1" });

const resp = await client.chat.completions.create({
  model: "grok-4.3",
  messages: [
    { role: "system", content: "You are a concise assistant." },
    { role: "user", content: "Name three benefits of private AI inference." },
  ],
  temperature: 0.7,
  max_tokens: 300,
});
console.log(resp.choices[0].message.content);
console.log(resp.usage);
```

Notice that the Python and TypeScript examples use the official OpenAI SDK pointed at Nito's base URL. That is the whole integration: Nito speaks the OpenAI chat shape, so for most apps the change is the `base_url` and the key, nothing more.

## What happens to this request

Before reading the fields, it helps to know the path the call takes, because it explains the response and the billing. When this request arrives, Nito authenticates the key and checks rate limits, reads the `model` field to resolve which model and privacy mode you want, routes to a healthy backend that can serve it, places a billing hold sized to the expected cost, runs the inference, and then settles the hold for the actual usage. You only write the request; the gateway does the rest. The full walkthrough is in [How Nito Works](/get-started/how-nito-works).

## What you sent

| Field | Type | Required | Meaning |
| --- | --- | --- | --- |
| `model` | string | optional | The model ID. If omitted, a default model is used. Add `:tee` or `:e2e-tee` to raise privacy. |
| `messages` | array | required | The conversation. Each item is `{ role, content }`. Roles: `system`, `user`, `assistant`, `tool`, `developer`. |
| `temperature` | number | optional | Sampling randomness. Higher is more varied. Passed through to the model. |
| `top_p` | number | optional | Nucleus sampling, an alternative to temperature. |
| `max_tokens` | integer | optional | Upper bound on generated tokens. |
| `stream` | boolean | optional | When `true`, stream tokens as Server-Sent Events. Defaults to `false`. |

A few things worth understanding rather than memorizing:

- **Roles** divide the conversation by speaker. `system` (or `developer`) sets instructions and persona, `user` is the human turn, `assistant` is a previous model turn you are replaying back, and `tool` carries the result of a tool the model asked to call. A `tool` message must reference the call it answers via `tool_call_id`.
- **The body is decoded strictly.** An unknown field returns `400` rather than being ignored. This is deliberate: a typo like `max_token` fails loudly instead of silently doing nothing.
- **Only `messages` is truly required.** `model` falls back to a default if you leave it out, though naming a model explicitly is recommended so your behavior does not change when defaults change.
- **`temperature`, `top_p`, and `max_tokens` are passthrough.** Nito forwards them to the model rather than reinterpreting them, so they behave as they do on the underlying model.

## What you got back

```json
{
  "id": "...",
  "object": "chat.completion",
  "created": 1751299200,
  "model": "grok-4.3",
  "choices": [
    {
      "index": 0,
      "message": {
        "role": "assistant",
        "content": "1. Your prompts stay private. 2. ... 3. ..."
      },
      "finish_reason": "stop"
    }
  ],
  "usage": {
    "prompt_tokens": 28,
    "completion_tokens": 64,
    "total_tokens": 92
  }
}
```

Reading it field by field:

- **`object`** is always `"chat.completion"` for a non-streaming chat reply, and **`created`** is the Unix timestamp (seconds) when it was produced.
- **`choices[0].message.content`** is the answer. There is always exactly one choice: Nito does not return multiple candidates, so you can always read `choices[0]`.
- **`finish_reason`** tells you why generation stopped. It is one of exactly four values: `stop` (natural end), `length` (hit `max_tokens`), `tool_calls` (the model wants to call a tool), or `error`.
- **`usage`** is what you were billed: `prompt_tokens` (input), `completion_tokens` (output), and `total_tokens`. A token is a chunk of text, very roughly a few characters, and it is the unit models are metered in. If prompt caching applied, you also see `cache_read_input_tokens` and `cache_write_input_tokens`.
- **`system_fingerprint`** is always present and currently always serialized as `null`. Do not rely on it.

## Add privacy with one change

To run the same request inside a Trusted Execution Environment (a TEE: a hardware-isolated, sealed compute box the host cannot inspect), add the `:tee` suffix to the model, provided that model supports it:

```json
{ "model": "grok-4.3:tee", "messages": [ ... ] }
```

Nothing else in the request changes. The privacy mode lives entirely in the model string, and if the chosen model cannot serve the mode you asked for, the call is rejected rather than quietly downgraded. After a `:tee` call you can fetch an attestation (a signed statement about the sealed environment that ran your call) and verify it yourself. See [TEE attestation](/privacy/tee-attestation) and the [attestation endpoint](/api-reference/attestation).

## Common mistakes

- **Empty `messages`.** At least one message is required, or you get `400 messages is required`.
- **An unknown field.** The decoder rejects unknown keys with `400`. Check for typos like `max_token` or `temprature`.
- **A `tool` message without `tool_call_id`.** Tool-result messages must reference the call they answer.
- **Assuming multiple choices.** Nito returns a single choice; there is no `n` fan-out on chat completions. To compare several models on one prompt, see [chat fusion](/features/multi-model-routing).
- **Reading a streamed response as one JSON object.** If you set `stream: true`, the reply is many Server-Sent Events, not one document. See the next page.
- **Expecting `:tee` to fall back silently.** A privacy mode never downgrades. If the model cannot serve it, the call fails instead.
