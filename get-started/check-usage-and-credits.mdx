After your first successful call, the natural next question is: did that cost anything, and how much do I have left? This page shows how to read your credit balance and your recent usage, and explains how Nito bills a request so the numbers make sense. The short answer is that Nito prepays each call against your balance and then settles for what the call actually used, so understanding the hold-then-settle flow is the key to reading every number on this page.

## How billing works, briefly

Nito bills in credits. Internally those credits are tracked in nanos, billionths of a credit, so that fractions of a cent's worth of tokens can be accounted for exactly without rounding error. You will see `nanos` in the API responses; it is just the precise unit underneath the human-facing credit.

The flow around a single call has three steps:

1. **Hold.** Before spending real money on a provider call, Nito places a hold against your balance, sized to the expected cost of the request. A hold is like the pre-authorization a hotel puts on your card at check-in: it reserves funds without charging them yet, which guarantees you can pay for the work before it starts.
2. **Settle.** When the call succeeds, the hold is captured for the actual usage, which may be less than the estimate. You are charged what the call really used.
3. **Release.** If the request fails before producing any usage, the hold is released and you are not charged. This is exactly why a failed call does not cost you credits.

Every successful response also includes a `usage` block (input tokens, output tokens, total). That is the per-call record. For account-level totals and balance, use the endpoints below.

## Why balance and usage are per organization

Billing in Nito is owned by the organization, not the individual user or the individual key. (An organization is the account that owns billing and keys.) That single fact explains the shape of every endpoint here: balance and usage both take an org ID in the path, and there is no flat, account-wide balance or usage URL. If you keep "billing lives on the org" in mind, the endpoint design is predictable.

You can find your org IDs from `GET /api/v1/orgs`.

## Check your balance

Read the balance from the org billing endpoint:

```bash
curl "https://YOUR-NITO-BASE-URL/api/v1/orgs/$ORG_ID/billing/balance" \
  -H "Authorization: Bearer $NITO_API_KEY"
```

```json
{ "available_nanos": "4250000000" }
```

`available_nanos` is your spendable balance in nanos, returned as a decimal string (not a number) to avoid precision loss in clients that would otherwise round large integers. To convert to whole credits, divide by 1,000,000,000. The example `"4250000000"` is 4.25 credits. An org that has no balance account yet returns `"0"`.

## Check recent usage

For a usage summary over a time window, use the org API overview:

```bash
curl "https://YOUR-NITO-BASE-URL/api/v1/orgs/$ORG_ID/api/overview?window_days=30" \
  -H "Authorization: Bearer $NITO_API_KEY"
```

This returns aggregated totals plus a daily and a per-model breakdown, including requests, input and output tokens, cached tokens, reasoning tokens, web-search counts, and the customer charge in nanos. The `window_days` parameter sets how far back the window reaches: it defaults to 30 and accepts up to 365.

Key fields:

- **`all_time`** and **`current_window`** totals: `requests`, `input_tokens`, `output_tokens`, `reasoning_tokens`, `total_tokens`, `customer_charge_nanos`, and more. Use `current_window` for "spend in the last N days" and `all_time` for the lifetime figure.
- **`daily[]`** per-day usage, including the top model that day.
- **`models[]`** per-model usage, sorted by charge, so the most expensive model is easy to find.

This overview is aggregated, not a transaction log. It is built for "how much did we use this month, and on what," not for listing individual requests.

## Failed-request billing

The hold-then-settle design above is what makes failed-request billing intuitive:

- If a call errors before the provider returns any usage, Nito voids the hold and you are not charged.
- If a streaming call fails after some usage has already been recorded, the charge is reconciled for the usage that actually occurred rather than dropped. You pay for what was produced, not for the part that failed.

See [Failed request billing behavior](/api-reference/usage-and-credits/failed-request-billing).

## Common mistakes

- **Calling a flat balance URL.** Balance is per org: `GET /api/v1/orgs/{orgId}/billing/balance`. There is no account-wide variant.
- **Treating nanos as whole credits.** `available_nanos` is in billionths. Divide by 1,000,000,000 (1e9) before showing it to a person.
- **Parsing the balance as a number.** It is a decimal string. Keep it as a string or a big-decimal type so you do not lose precision.
- **Expecting a real-time per-request ledger endpoint.** Use the response `usage` block for each call and the org overview for aggregates. There is no public per-transaction history endpoint.
- **Expecting `X-RateLimit-*` headers.** Usage and balance reads have their own rate limits, and a `429` carries only `Retry-After`.

## Notes

- Retention: usage aggregates are derived from billing records, not from your prompt contents. Reading usage does not expose what was in any prompt.
