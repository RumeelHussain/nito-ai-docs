Enterprises often ask for three things together: **bring-your-own-key (BYOK)** so the provider never sees a raw model key, an **attested** execution environment, and **audited, governed debug capture** for incident review. This recipe is honest about which of those you can build on Nito today and which you cannot.

**What is buildable today:** attested inference on the `:confidential` tier, with attestation you can fetch and verify. **What is not available yet:** BYOK and audited debug capture. Those are marked below with SME TODOs and belong on the deferred list; do not build against them until confirmed.

## The buildable part: attested (Confidential / `:confidential`) inference

The attested tier is real and shippable. Select it with the `:confidential` model-ID suffix on an attestation-capable model (NEAR or Phala in the current catalog). Fetch `GET /v1/models` and pick a model whose card advertises attestation. The example model `grok-4.3` is a placeholder for a non-attested tier and does **not** support `:confidential`.

```bash
curl -sS "https://YOUR-NITO-BASE-URL/v1/chat/completions" \
  -H "Authorization: Bearer $NITO_API_KEY" \
  -H "Content-Type: application/json" \
  -d '{
    "model": "MODEL_ID:confidential",
    "messages": [{"role": "user", "content": "Classify this internal ticket."}]
  }'
```

Then prove the tier rather than trust it. Fetch `GET /api/v1/attestation?model=...&nonce=<fresh>` and run the baseline checks (status 200, `object` is `attestation`, `privacy_tier` is `confidential`, `attestation_available` true, the nonce echo matches, evidence present, proof metadata present, `verified` true). The gateway does **baseline** verification only; validating the Intel TDX quote, GPU evidence, and certificate chains in `evidence` is your responsibility. The full flow, with a runnable checker, is in [TEE-attested call and verify nonce](/cookbook/private-ai/tee-attested-call-verify-nonce).

Alongside the attested tier, the gateway's baseline privacy posture applies: metadata-only retention (prompts, completions, reasoning, tool, and attachment payloads are never persisted) and `Cache-Control: no-store` on inference responses.

This attested-plus-verification core is enough to stand up confidential inference for many enterprise workloads today.

## Not yet available: BYOK

BYOK is **not implemented in the gateway.** Provider keys come from gateway configuration only; there is no path for a customer to supply their own upstream provider key per request or per org. Do not design a flow that assumes customer-supplied provider keys, and do not present one to stakeholders as available.

## Not yet available: audited / governed debug capture

An audited, governed debug-capture subsystem (where sensitive request/response payloads are retained under governance for a bounded window and access is logged) is **not confirmed in code.** The privacy-and-retention doc mentions "bounded retention" without a number, and no debug-capture subsystem or retention constant was found. The commonly cited "up to 7 days" figure is **not confirmed** and must not be stated as fact.

## What this means for the recipe title

Because BYOK and audited debug capture are not available, this recipe **cannot be delivered end to end as titled today.** Ship the attested-inference core now, and treat the two missing pieces as deferred until the owners above confirm them. If a customer requires BYOK or governed debug capture as a hard prerequisite, that requirement is currently unmet; say so rather than approximating it.
