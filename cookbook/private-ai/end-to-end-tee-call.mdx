The `:e2e-tee` mode is the strictest privacy tier on Nito. Where `:tee` gives you an attested route, `:e2e-tee` adds end-to-end encryption between your client and the model inside the trusted execution environment, negotiated through a set of `X-Z-TEE-*` request headers. This recipe covers how to select the mode and the constraints it imposes. The exact encryption handshake is owner-confirmed work, marked below.

## What `:e2e-tee` requires and forbids

Selecting end-to-end TEE is done with the `:e2e-tee` model-ID suffix, and it only works on **NEAR or Phala** models (the attestation-capable providers). It is more constrained than an ordinary or `:tee` call:

- **Non-streaming only.** With the E2EE headers present, streaming is refused (`streaming is not supported with E2E-TEE`). Do not set `stream: true`.
- **No tools, reasoning, prompt caching, or multi-part messages.** Each of these is rejected when the E2EE headers are present. Send a plain, single-part message request.
- **NEAR / Phala only.** Other providers reject the mode.
- **Requires the `X-Z-TEE-*` headers.** The mode is not active on the suffix alone; the encryption headers must be present and well-formed.

## The required headers

An `:e2e-tee` request carries this set of headers, parsed by the gateway into its E2EE encryption context:

- `X-Z-TEE-E2EE`
- `X-Z-TEE-Client-Pub-Key`
- `X-Z-TEE-Model-Pub-Key`
- `X-Z-TEE-Signing-Algo`
- `X-Z-TEE-Encryption-Version`
- `X-Z-TEE-E2EE-Nonce`
- `X-Z-TEE-E2EE-Timestamp`

The model public key you place in `X-Z-TEE-Model-Pub-Key` is the same model public key returned by the attestation endpoint (`signing_key` / `model_public_key` are aliases for it). So a typical flow is: fetch the attestation for the `:e2e-tee` model, take the model public key from it, perform the key exchange, and set these headers on the chat call.

## When to use `:e2e-tee` vs `:tee`

- Use `:tee` when you want an **attested** route and still need streaming, tools, reasoning, or caching. Verify it with the [attested-call recipe](/cookbook/private-ai/tee-attested-call-verify-nonce).
- Use `:e2e-tee` when the request and response payloads must be **encrypted end to end** to the enclave and you can accept the non-streaming, no-tools, single-part constraints.

Either way, attestation is still how you gain assurance: fetch and verify the attestation for the model before trusting the tier. See [Verify it yourself](/privacy/tee-attestation/verify-yourself-recipe).
