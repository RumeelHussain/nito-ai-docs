If you already have an app talking to an OpenAI-compatible endpoint, you do not need to rewrite it to get a stronger privacy posture on Nito. Privacy is selected by the **model ID**, not by a request field or a custom parameter. That means the upgrade is a one-line change per call: keep your existing request shape, and change the model you ask for.

This recipe shows the smallest possible diff to move an existing app onto a private or attested tier.

## How privacy selection works

Nito exposes privacy as a **model-ID suffix**. A logical model ID is `<model>`, `<model>:confidential`, or `<model>:encrypted`, and no other suffix is accepted. There is no `:tier` parameter and no request-body field that turns privacy on. So the whole upgrade is choosing the right model string.

Two ways to raise privacy:

1. **Pick a private-tier model.** Some models are stored at the `private` tier, which routes with zero-data-retention set on the upstream. You select these simply by using that model's ID.
2. **Add the `:confidential` suffix.** For a model whose provider supports a trusted execution environment, appending `:confidential` requests the attested `confidential` tier.

Not every model supports every tier. The `:confidential` suffix only works on attestation-capable models (NEAR or Phala providers in the current catalog). If you add `:confidential` to a model that does not support it, the request is rejected. **Check the catalog first** by fetching `GET /v1/models` and choosing a model whose card advertises the tier you want.

## The one-line change

Here is a typical existing call. The example model `grok-4.3` is a placeholder; fetch `GET /v1/models` to get a real model ID for your account.

```bash
curl -sS "https://YOUR-NITO-BASE-URL/v1/chat/completions" \
  -H "Authorization: Bearer $NITO_API_KEY" \
  -H "Content-Type: application/json" \
  -d '{
    "model": "grok-4.3",
    "messages": [{"role": "user", "content": "Summarize this contract clause."}]
  }'
```

To upgrade, change only the `model` value. For an attested call, append `:confidential` to an attestation-capable model:

```bash
curl -sS "https://YOUR-NITO-BASE-URL/v1/chat/completions" \
  -H "Authorization: Bearer $NITO_API_KEY" \
  -H "Content-Type: application/json" \
  -d '{
    "model": "MODEL_ID:confidential",
    "messages": [{"role": "user", "content": "Summarize this contract clause."}]
  }'
```

Everything else stays the same: the same `/v1/chat/completions` path, the same `messages` shape, the same auth header, the same streaming behavior. Your response envelope is unchanged (`object: "chat.completion"`, one `choices` entry, `usage`).

## What changes and what does not

- **Retention:** the gateway is metadata-only by default. Prompts, completions, reasoning, tool payloads, and attachments are not persisted, and inference responses carry `Cache-Control: no-store`. A private or attested tier keeps that posture and, for `:confidential`, adds an attestation you can fetch and verify.
- **Feature compatibility on `:confidential`:** an attested chat call still supports streaming and tools. (The stricter `:encrypted` mode is different: it is non-streaming and rejects tools, reasoning, prompt caching, and multi-part messages. See [Encrypted call](/cookbook/private-ai/encrypted-call).)
- **Not a hardware proof by itself:** choosing `:confidential` gives you an attested route. To confirm the attestation, fetch and check it. That is a separate, small step covered in [TEE-attested call and verify nonce](/cookbook/private-ai/tee-attested-call-verify-nonce).

## Rollout tip

Because the only thing that changes is the model string, you can gate the upgrade behind config. Read the model ID from an environment variable and flip it per environment (for example, `grok-4.3` in dev and `MODEL_ID:confidential` in production), so the privacy upgrade ships without touching request-building code.
