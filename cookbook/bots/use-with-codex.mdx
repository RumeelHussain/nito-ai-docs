Codex is an OpenAI-oriented coding tool. You can route it through Nito because Nito exposes an OpenAI-compatible API. You override its base URL, key, and model rather than selecting a named "Nito" provider.

This page is a recipe pointer. The full, maintained setup lives in the [Codex editor integration](/sdks/editor-integrations/codex).

## The three values

Wherever Codex accepts an OpenAI-compatible base URL and key, supply:

- **Base URL:** `https://YOUR-NITO-BASE-URL/v1`. The path must end in `/v1`, because OpenAI-style clients append `/chat/completions` themselves.
- **API key:** your Nito key, which starts with `z_sk_`.
- **Model:** a Nito catalog ID such as `grok-4.3`. Do not use an OpenAI name like `gpt-4o`.

Fetch a real model ID from the live catalog:

```bash
curl https://YOUR-NITO-BASE-URL/v1/models \
  -H "Authorization: Bearer z_sk_your_key_here"
```

## Use Chat Completions, not Responses

Nito's compatible surface is OpenAI **Chat Completions** at `POST /v1/chat/completions`. The OpenAI **Responses API** (`/v1/responses`) is not served and returns 404. Newer OpenAI-oriented tools sometimes default to the Responses API, so confirm that your Codex version can target Chat Completions before relying on this route. If Codex only speaks the Responses API in your version, it will not reach Nito.

## Notes

- Privacy is selected by the model ID. Append `:tee` for a TEE-backed run with a verifiable attestation, for example `grok-4.3:tee`. `:e2e-tee` does not stream and needs extra headers, so it is generally not usable from a coding tool's request loop. See [Per-call privacy modes](/privacy/per-call-privacy-modes).
- For the current settings path and troubleshooting, follow the [Codex editor integration](/sdks/editor-integrations/codex) page.
