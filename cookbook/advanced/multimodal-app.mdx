**Goal.** A small app that combines three multimodal steps: read an image in a chat request, parse a document into a prompt, and generate an image from text.

<Warning>
Two of the three steps below don't work with an API key right now (verified live):
- **Image generation (Step 3)** — the image backend isn't serving: `POST /v1/images/generations` returns `400 "model is not available"` even with a prompt-only request. Raised with the team.
- **Document parsing (Step 2)** — `POST /api/v1/attachments/parse` is **browser-only by design** (confirmed with the team): it returns `403 "browser session required"` for `z_sk_` keys, so it can't be called with an API key.

**Image input (Step 1) does work** — just pass the image as **inline base64 / a data URL**, not an arbitrary URL. The gateway only accepts images from approved hosts, so a random `image_url` returns `400 "host is not allowed"`.
</Warning>

**Prerequisites.** An organization key in `NITO_API_KEY`, your base URL, and a chat model that declares image support (`supportsImageInputs` in `GET /v1/models`).

## Read this first: what is and is not supported

Honesty up front, because it is easy to assume a modern gateway does everything. Nito's multimodal surface is exactly three things:

- **Image input** (PNG, JPEG, WebP) into a chat model.
- **Document input** (PDF, DOCX, PPTX, XLSX, XLS, plain text), parsed to text server-side.
- **Image generation** (text to a PNG image).

**Audio and video are not available**, as input or as generation. There are no audio or video endpoints today. Do not design a step around them. See [Multimodal](/features/multimodal) and [Known limitations](/privacy/known-limitations) for the full boundary. This recipe builds only on the three capabilities above.

## Step 1: Image input into chat

Send an image as a content part alongside your text. Formats are PNG, JPEG, WebP. You can pass a URL, a data URL, or raw base64 with a media type.

```python
import os, json, urllib.request

BASE_URL = "https://nito.dev.zprotocol.org"   # TODO: confirm canonical base URL
API_KEY = os.environ["NITO_API_KEY"]

def post(path, body):
    req = urllib.request.Request(
        f"{BASE_URL}{path}", data=json.dumps(body).encode(),
        headers={"Authorization": f"Bearer {API_KEY}",
                 "Content-Type": "application/json"},
    )
    with urllib.request.urlopen(req) as resp:
        return json.load(resp)

import base64
with open("photo.png", "rb") as f:                 # a local image file
    b64 = base64.b64encode(f.read()).decode()

vision = post("/v1/chat/completions", {
    "model": "x-ai/grok-4.5",   # must declare image support; check GET /v1/models
    "messages": [{
        "role": "user",
        "content": [
            {"type": "text", "text": "What is in this image?"},
            # pass the image inline as a data URL — an arbitrary URL is rejected ("host is not allowed")
            {"type": "image_url",
             "image_url": {"url": f"data:image/png;base64,{b64}"}},
        ],
    }],
})
print(vision["choices"][0]["message"]["content"])
```

`x-ai/grok-4.5` is a stand-in; fetch a real image-capable ID from `GET /v1/models`. Limits: up to 10 images per request, 5 MiB per image, 2048px on the long edge.

## Step 2: Parse a document into a prompt

A document does not go straight into chat; you parse it first at the parser endpoint, which is multipart. Then you put the returned text into a normal chat message.

```python
import urllib.request

def parse_document(file_path):
    # multipart upload to the parser endpoint
    import mimetypes, uuid
    boundary = uuid.uuid4().hex
    with open(file_path, "rb") as f:
        content = f.read()
    parts = (
        f"--{boundary}\r\n"
        f'Content-Disposition: form-data; name="file"; filename="{file_path}"\r\n'
        f"Content-Type: application/octet-stream\r\n\r\n"
    ).encode() + content + f"\r\n--{boundary}--\r\n".encode()
    req = urllib.request.Request(
        f"{BASE_URL}/api/v1/attachments/parse", data=parts,
        headers={"Authorization": f"Bearer {API_KEY}",
                 "Content-Type": f"multipart/form-data; boundary={boundary}"},
    )
    with urllib.request.urlopen(req) as resp:
        return json.load(resp)

parsed = parse_document("contract.pdf")
answer = post("/v1/chat/completions", {
    "model": "x-ai/grok-4.5",
    "messages": [{"role": "user",
                  "content": f"Summarize this document:\n\n{parsed['text']}"}],
})
print(answer["choices"][0]["message"]["content"])
```

The parser accepts PDF, DOCX, PPTX, XLSX, XLS, and text; it **rejects images** (send those directly to an image-capable model as in Step 1). It returns `text`, `mime_type`, `file_kind`, `bytes`, and optional `page_count`, and caps output at 300k runes and PDFs at 80 pages. The upload cap is 10 MiB.

## Step 3: Generate an image

Generation is a separate endpoint with its own request shape. You send a prompt, you get PNG data back.

```python
image = post("/v1/images/generations", {
    "prompt": "a diagram of a sealed compute enclave, flat vector style",
    "n": 1,
    "size": "1024x1024",
})
print(image["data"][0].get("url") or "b64 image returned")
```

Response format is a `url` by default, or `b64_json` if you ask. Output is always PNG. `n` is 0 to 4.

## Putting it together

A realistic flow chains all three: parse an uploaded document (Step 2), ask a chat model about it and about a supplied image (Steps 1 and 2 combined in one call, since one chat request can mix text, image inputs, and document text), then generate an illustration for the result (Step 3). That is the full extent of multimodal on Nito today. When you find yourself reaching for an audio transcription or a video step, stop: those are not available, and the honest move is to handle them outside Nito rather than assume an endpoint that does not exist.
