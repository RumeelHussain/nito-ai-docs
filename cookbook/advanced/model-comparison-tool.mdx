**Goal.** A small script that discovers the available chat models, sends the same prompt to a handful of them, and lays the answers out side by side so you can judge quality, tone, and latency for your own task.

**Prerequisites.** An organization key (`z_sk_...`) in `NITO_API_KEY`, your base URL, and Python 3 with the standard library. If the model IDs are new, read [How Nito works](/get-started/how-nito-works) first.

Because you reach every model through one key and one base URL, comparing models is just a loop over model IDs, not a loop over provider integrations. That is the whole reason this recipe is short.

## Step 1: Discover the models

Never hard-code a model name. Ask the gateway what exists and pick from the live catalog.

```bash
curl -sS "https://YOUR-NITO-BASE-URL/v1/models" \
  -H "Authorization: Bearer $NITO_API_KEY"
```

Each entry has an `id` (what you pass back in a request), a `type` (`chat`, `embedding`, `image`), its `capabilities`, and its `privacy_tier`. Filter to `type == "chat"` and take the IDs you want to compare.

## Step 2: Send one prompt to several models

The example below uses `grok-4.3` as a stand-in; substitute IDs you pulled from `GET /v1/models`.

```python
import os, json, time, urllib.request

BASE_URL = "https://YOUR-NITO-BASE-URL"   # TODO: confirm canonical base URL
API_KEY = os.environ["NITO_API_KEY"]
MODELS = ["grok-4.3"]  # add more IDs fetched from GET /v1/models

PROMPT = "Explain what a Trusted Execution Environment is, in two sentences."

def ask(model):
    body = json.dumps({
        "model": model,
        "messages": [{"role": "user", "content": PROMPT}],
    }).encode()
    req = urllib.request.Request(
        f"{BASE_URL}/v1/chat/completions", data=body,
        headers={"Authorization": f"Bearer {API_KEY}",
                 "Content-Type": "application/json"},
    )
    started = time.time()
    with urllib.request.urlopen(req) as resp:
        data = json.load(resp)
    latency_ms = int((time.time() - started) * 1000)
    return {
        "model": model,
        "request_id": resp.headers.get("X-Request-ID"),
        "text": data["choices"][0]["message"]["content"],
        "usage": data.get("usage", {}),
        "latency_ms": latency_ms,
    }

for model in MODELS:
    r = ask(model)
    print(f"\n=== {r['model']}  ({r['latency_ms']} ms, req {r['request_id']}) ===")
    print(r["text"])
    print("tokens:", r["usage"].get("total_tokens"))
```

Every response carries an `X-Request-ID` header, so you can capture it alongside each answer for later debugging. The `usage` block gives you `prompt_tokens`, `completion_tokens`, and `total_tokens` to compare cost and verbosity.

## Step 3: Compare on the axes that matter

Print the three things that separate models on a real task: the text quality (your judgment), the token count (a proxy for cost and verbosity), and the latency you measured. Run the loop a few times, because latency varies with routing and load.

## The gateway-native way: fusion

If your goal is not just to compare but to merge, Nito has a built-in endpoint for that: `POST /api/v1/chat/fusion`. It asks 2 to 3 distinct models the same prompt concurrently, then runs one synthesis (judge) pass that merges the answers into a single result.

```python
body = json.dumps({
    "models": ["grok-4.3", "another-model-id"],   # 2 to 3 distinct IDs
    "messages": [{"role": "user", "content": PROMPT}],
}).encode()
req = urllib.request.Request(
    f"{BASE_URL}/api/v1/chat/fusion", data=body,
    headers={"Authorization": f"Bearer {API_KEY}",
             "Content-Type": "application/json"},
)
with urllib.request.urlopen(req) as resp:
    fusion = json.load(resp)

print("synthesis:\n", fusion["synthesis"])
for m in fusion["models"]:
    print(m["model"], m["latency_ms"], "ms", m.get("finish_reason"))
```

A few honest constraints. Fusion is **non-streaming and batched only**: you get one complete response, not a token stream. It takes **2 to 3 distinct models**. The response object is `"fusion.completion"` and includes the `judge` used, the merged `synthesis`, per-model results in `models[]` (each with `output`, `tokens`, `latency_ms`, `finish_reason`), and synthesis token counts. Only hosted web tools are allowed; caller-defined function tools are rejected. Privacy is chosen per model via the ID suffix, but you cannot mix TEE and non-TEE models in one call, and `:e2e-tee` is rejected for fusion.

Use your own loop (Steps 1 to 3) when you want raw side-by-side answers and streaming; use fusion when you want the gateway to fan out and merge for you in one call.
