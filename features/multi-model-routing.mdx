description: Ask two or three models the same prompt at once, then have a judge model merge their answers into one. A separate, non-streaming Nito endpoint.
---

Ordinary chat picks one model to answer your prompt. Fusion does something different: it asks **two or three models the same prompt at the same time**, then runs a single extra pass where a "judge" model reads all their answers and synthesizes one merged reply. You get back both the merged answer and every participant's raw output, so you can see where the models agreed, where they diverged, and what the judge did with the difference. This is useful when an answer matters enough to be worth more than one model's opinion, and when you would rather have the gateway run the ensemble than orchestrate it yourself.

This page assumes you already understand ordinary [Chat Completions](/features/chat-completions) and that the `model` string carries both a model and a privacy mode. Fusion builds directly on both ideas.

## How fusion works

The flow is two stages:

1. **Fan-out.** The gateway sends your prompt to each model in `models[]` concurrently. They run in parallel, not one after another, so total latency is roughly the slowest participant plus the synthesis pass, not the sum of all of them.
2. **Synthesis (the judge).** Once the participants return, a single judge model reads all of their outputs and produces one merged answer. The response reports which model acted as judge and the synthesized text.

Fusion is **non-streaming and batched only.** There is no token-by-token stream; you get the complete result in one response when the whole process finishes. Each participant runs under a budget of roughly 150 seconds.

## The endpoint

Fusion lives at a single first-party path:

```
POST /api/v1/chat/fusion
```

There is **no `/v1` OpenAI-compatible alias** for fusion. This is a first-party Nito capability, so use the `/api/v1` path only. Pointing an OpenAI SDK at a `/v1/chat/fusion` path will not work.

## The request

The body is `apiFusionRequest`. It looks like a chat request with a `models[]` array in place of a single `model`.

| Field | Type | Notes |
|---|---|---|
| `messages` | array | **Required.** The conversation, same shape as chat completions. |
| `models` | array of strings | **Required. 2 to 3 distinct models** (minimum 2, maximum 3). These are the participants asked in parallel. |
| `temperature` | number | Optional, passed through to participants. |
| `top_p` | number | Optional, passed through to participants. |
| `max_tokens` | integer | Optional cap on generated tokens. |
| `user` | string | Optional opaque end-user identifier for your bookkeeping. |
| `web_search` | bool or object | Optional. Let participants search the web mid-generation. |
| `web_fetch` | bool or object | Optional. Let participants fetch a URL mid-generation. |
| `tools` | array | Optional, **hosted web tools only.** Function (caller-defined) tools are rejected. |

Two constraints are worth stating loudly because they cause rejections:

- **Only hosted web tools are allowed.** You can enable `web_search` and `web_fetch`, but a function tool in `tools` is rejected. Fusion is not a tool-execution loop; it is an ensemble plus a judge.
- **`models` must be 2 or 3 distinct entries.** Fewer than 2 or more than 3, or duplicates, are rejected.

### Privacy and the TEE constraint

Just like ordinary chat, privacy rides on the **model suffix** of each entry in `models[]`. But fusion adds two rules:

- **You cannot mix TEE and non-TEE models** in the same fusion request. Either all participants are attested-tier or none are; a mixed set is rejected.
- **`:e2e-tee` is rejected for fusion.** End-to-end encrypted TEE is non-streaming and deliberately feature-limited, and it is not available as a fusion participant.

The `:tee` suffix is allowed, in which case the response can carry attestations (see below).

## The response

The body is `apiFusionResponse` with `object` set to `"fusion.completion"`.

| Field | Notes |
|---|---|
| `object` | Always `"fusion.completion"`. |
| `judge` | The model that performed the synthesis pass. |
| `synthesis` | The merged answer text. This is the main output. |
| `synthesis_prompt_tokens` | Prompt tokens consumed by the synthesis pass. |
| `synthesis_tokens` | Tokens generated by the synthesis pass. |
| `synthesis_attestation` | Optional. The attestation for the synthesis pass, when it ran in an attested environment. |
| `models` | Array of per-participant results (see below). |
| `free_tier` | Optional free-tier accounting, when applicable. |

Each entry in `models` is an `apiFusionModelResult`:

| Field | Notes |
|---|---|
| `id` | The participant's result identifier. |
| `model` | The model that produced this result. |
| `provider` | The provider that served it. |
| `output` | That model's raw answer to the prompt. |
| `prompt_tokens` | Prompt tokens for this participant. |
| `tokens` | Generated tokens for this participant. |
| `latency_ms` | How long this participant took, in milliseconds. |
| `finish_reason` | Why this participant stopped. |
| `web_search` | Optional web-search accounting, when the participant searched. |
| `attestation` | Optional attestation, when this participant ran in an attested environment. |
| `error` | Optional. Set when this particular participant failed. |

### Partial failures

Fusion is built to degrade gracefully. If one or two participants fail but at least one succeeds, the request still returns **`200`**, the failed participant carries an `error` in its `models[]` entry, and the judge synthesizes from whatever did succeed. Only when **all** participants fail does the request return an error instead of a `200`. So always check each `models[]` entry for an `error`, even on a successful response.

The model IDs below are illustrative. Fetch `GET /v1/models` for the live catalog and select participants by capability at runtime, because models are added and retired over time.

```bash
curl https://YOUR-NITO-BASE-URL/api/v1/chat/fusion \
  -H "Authorization: Bearer $NITO_API_KEY" \
  -H "Content-Type: application/json" \
  -d '{
    "models": ["grok-4.3", "model-b", "model-c"],
    "messages": [
      {"role": "system", "content": "You are precise and cite trade-offs."},
      {"role": "user", "content": "What are the trade-offs of running inference inside a TEE?"}
    ],
    "max_tokens": 800
  }'
```

A successful response, abbreviated:

```json
{
  "object": "fusion.completion",
  "judge": "grok-4.3",
  "synthesis": "Running inference in a TEE trades some latency and availability for a verifiable privacy boundary...",
  "synthesis_prompt_tokens": 412,
  "synthesis_tokens": 256,
  "models": [
    {
      "id": "...",
      "model": "grok-4.3",
      "provider": "...",
      "output": "A TEE isolates the computation in sealed hardware...",
      "prompt_tokens": 38,
      "tokens": 190,
      "latency_ms": 2210,
      "finish_reason": "stop"
    },
    {
      "id": "...",
      "model": "model-b",
      "provider": "...",
      "output": "The main cost is reduced backend availability...",
      "prompt_tokens": 38,
      "tokens": 175,
      "latency_ms": 2640,
      "finish_reason": "stop"
    },
    {
      "id": "...",
      "model": "model-c",
      "error": "provider unavailable"
    }
  ]
}
```

Here `model-c` failed, but the call still returned `200` and the judge synthesized from the two that succeeded. The `synthesis` field is the answer most callers use; the `models[]` array is there when you want to inspect each participant or audit the merge.

## When to reach for fusion

Use fusion when the extra cost of running several models is worth it: high-stakes answers, questions where models are known to disagree, or cases where you want to surface and reconcile disagreement rather than trust one model. For everyday text, ordinary [Chat Completions](/features/chat-completions) is cheaper and supports streaming. Remember that fusion bills for every participant plus the synthesis pass, and the same [idempotency](/features/idempotency) rules apply, with each subcall keyed deterministically from your base key.
