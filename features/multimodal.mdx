Most of what you send Nito is text, but the gateway is not limited to text. You can put images into a chat request and let a vision-capable model read them, you can attach documents (including PDFs) and have them parsed into text the model can reason over, and you can ask a model to generate an image from a prompt. This page is the map of what is and is not multimodal on Nito, so you know the edges before you build against them.

## The Three Things that Work

There are exactly three multimodal capabilities, two on the way in and one on the way out.

- **Image input.** You send one or more images inside a chat message and a vision-capable model reads them alongside your text. Supported formats are PNG, JPEG, and WebP. You can pass an image as a URL, as a data URL, or as raw base64 with a media type that the gateway assembles for you. The model you route to must declare image support. Full detail, limits, and a worked example are on [Image input](/features/multimodal/image-input).
- **Document input.** You attach a document and the gateway parses it into text server-side before the model sees it. Supported kinds are PDF, DOCX, PPTX, XLSX, XLS, and plain text. This is how you get a contract, a slide deck, or a spreadsheet into a prompt. Detail, parsing caps, and the two ways to do it are on [PDF and document input](/features/multimodal/pdf-input).
- **Image generation.** You send a prompt and get back a PNG image. This is the only generation modality. Detail, parameters, and a worked example are on [Image generation](/features/multimodal/image-generation).

That is the whole surface. One model call can mix text with image inputs and document inputs; image generation is a separate endpoint with its own request shape.

## Input Versus Generation

It helps to keep two ideas apart, because they use different endpoints and different model types.

**Input** is what you feed a chat model so it can answer. Images and documents are inputs. They travel inside the `messages` array of a chat completion request as content parts, and they are handled by a chat model that supports them. Nothing is generated; the model reads your material and responds with text.

**Generation** is what a model produces for you. The only thing Nito can generate beyond text is an image, and that happens at a dedicated image endpoint, not inside a chat request. You send a prompt, you get image data back.

So "multimodal" on Nito means: rich inputs into chat (images, documents), plus one rich output (images) from a separate endpoint.

## How the Inputs Travel

Image input and document input both ride inside a normal chat request, as entries in a message's `content` array. Instead of a plain string, `content` becomes a list of typed parts: a `text` part for your words, an `image_url` part for a picture, a `file` part for a document. The model reads them together. The two sub-pages below show the exact shapes.

Document input has a second path worth knowing about: a standalone parser endpoint that turns a file into text you can then place into any prompt yourself. That is covered on the document page and in the [file parser](/api-reference/server-tools/file-parser) feature.
