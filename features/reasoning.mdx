title: "Reasoning"
description: "Ask a capable model to think before it answers. Control reasoning effort and summaries, read reasoning output, and budget for the extra tokens."
---

Some models can think before they answer: they work through a problem internally, then produce a final reply informed by that work. On Nito you turn this on with the `reasoning` field on a chat-completions request, and you control how much effort the model spends and how much of its thinking it shows you. Reasoning trades tokens (and therefore cost and latency) for quality on hard problems. This page covers when to use it, how to configure it, how to read the output, and what it costs.

Reasoning is a **per-model** capability. Not every model supports it. Check the model's `supports_reasoning` capability in `GET /v1/models` before relying on it, and select by capability at runtime rather than assuming.

## Turning it on

Add a `reasoning` object to your chat request:

```json
{
  "model": "grok-4.3",
  "messages": [
    {"role": "user", "content": "A bat and a ball cost $1.10. The bat costs $1.00 more than the ball. How much is the ball?"}
  ],
  "reasoning": { "enabled": true, "effort": "medium" }
}
```

The fields on `reasoning`:

| Field | Values | Meaning |
| --- | --- | --- |
| `enabled` | boolean | Turn reasoning on. |
| `effort` | `low`, `medium`, `high` | How hard the model thinks. More effort means more reasoning tokens. |
| `summary` | `auto`, `concise`, `detailed` | How much of the model's thinking is summarized back to you. |
| `max_tokens` | integer | A budget cap on reasoning tokens. |
| `exclude` | boolean | Run reasoning but omit the reasoning output from the response. |

Two defaults are worth remembering. If you enable reasoning without specifying `effort` or a token budget, effort defaults to **`low`**. If you do not specify `summary`, it defaults to **`auto`**, which lets the model choose an appropriate level of detail.

### Setting effort with a model-ID suffix

Reasoning can also be requested through a feature suffix on the model ID, the same mechanism that selects privacy modes. This is handy when you cannot easily set a request body field but can change the model string. The `reasoning` object and the suffix express the same capability; prefer the explicit object when you have it.

## Reading the output

When reasoning runs, the response carries two extra signals.

**`choices[].message.reasoning_details`** is an array describing the model's thinking. Each entry has a `type`, a `summary` (the human-readable account of that piece of reasoning), an `id`, a `format`, and an `index`. This is a summary of the reasoning, governed by the `summary` setting, not necessarily a verbatim transcript of every internal token.

**`usage.reasoning_tokens`** reports how many tokens were spent thinking. This is separate from `completion_tokens`, and it is what makes reasoning cost more.

```json
{
  "choices": [
    {
      "index": 0,
      "message": {
        "role": "assistant",
        "content": "Assume sqrt(2) = a/b in lowest terms ...",
        "reasoning_details": [
          { "type": "reasoning", "summary": "Set up a proof by contradiction.", "id": "rd_1", "format": "text", "index": 0 }
        ]
      },
      "finish_reason": "stop"
    }
  ],
  "usage": {
    "prompt_tokens": 18,
    "completion_tokens": 140,
    "reasoning_tokens": 320,
    "total_tokens": 478
  }
}
```

If you set `exclude: true`, the model still reasons (and you still pay for the reasoning tokens), but `reasoning_details` is omitted from the response. Use this when you want the quality benefit but do not want to handle or display the thinking.

When you stream, reasoning arrives incrementally: chunks carry `reasoning_details` deltas with a non-empty `summary` as the thinking unfolds, ahead of the final content. See [Streaming](/features/streaming).

## Cost and latency

Reasoning is not free. Reasoning tokens are billed, and `usage.reasoning_tokens` is the count you pay for on top of your prompt and completion tokens. Higher `effort` produces more reasoning tokens, so `high` costs more and takes longer than `low`. Treat effort as a dial: spend it on genuinely hard problems (math, multi-step logic, careful analysis) and leave it off for routine generation where it adds cost without improving the answer. Use `max_tokens` on the `reasoning` object to put a ceiling on how much the model can spend thinking.

## Privacy interaction

Reasoning works in ordinary, private, and `:tee` modes. The end-to-end encrypted TEE mode (`:e2e-tee`) **rejects reasoning**, along with streaming, tools, and prompt caching. If your call needs to think and needs the strongest privacy mode, those two requirements conflict today; pick the privacy tier that fits and use a reasoning-capable model within it. See [Per-call privacy modes](/privacy/per-call-privacy-modes).
