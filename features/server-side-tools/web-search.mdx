Web search lets a model pull fresh information from the web into its answer without you running anything. You set one field on your chat request, and the gateway decides whether a search is actually warranted, runs it through a native pipeline, folds the results into the model's context, and tells you in the response what it did. This page covers how to turn it on, every option you can pass, how the gateway decides whether to search, where it runs, and how it is billed.

## Turning On Web Search Feature

The simplest form is a boolean on the top-level `web_search` field:

```json wrap
{
  "model": "x-ai/grok-4.5",
  "messages": [{ "role": "user", "content": "What are the headline features of the latest release?" }],
  "web_search": true
}
```

`web_search` accepts either a boolean or an object. The object form unlocks the controls below. You can also enable search as a hosted tool entry, which is equivalent for turning the capability on:

```json wrap
{
  "tools": [{ "type": "web_search" }]
}
```

The chat request is decoded strictly: unknown fields inside the `web_search` object are rejected, and enabling the same feature in two conflicting places returns `request body contains duplicate feature fields`. Use one shape.

## Options

When `web_search` is an object, these fields are available.

| Field | Type | Default | Notes |
| --- | :-: | :-: | --- |
| `mode` | string |  | `on`, `off`, or `auto`. `auto` lets the rules engine and router decide. |
| `retrieval_mode` | string | `llm_context` | One of `llm_context` (default), `web_search`, or `web_fetch`. |
| `max_sources` | int | `10` | Maximum sources to retrieve. Capped at `50`. |
| `max_context_tokens` | int | `6000` | Token budget for search context folded into the prompt. |
| `freshness` | string |  | Recency constraint on results. |
| `country` | string |  | Country bias for results. |
| `search_lang` | string |  | Language for the search. |
| `safesearch` | string |  | Safe-search level. |
| `location` | object |  | Geographic hint (see below). |

The `location` object accepts `city`, `state`, `country`, `timezone`, `postal_code`, `latitude`, and `longitude`. Use it to localize results when the query is location-sensitive.

```json wrap
{
  "web_search": {
    "mode": "auto",
    "retrieval_mode": "llm_context",
    "max_sources": 8,
    "max_context_tokens": 4000,
    "freshness": "pw",
    "country": "US",
    "search_lang": "en",
    "location": {
      "city": "San Francisco",
      "state": "CA",
      "country": "US",
      "timezone": "America/Los_Angeles"
    }
  }
}
```

## How the Gateway Decides Whether to Search

Enabling `web_search` does not guarantee a search runs. The gateway uses a native pipeline with a rules engine that decides, per request, whether searching would actually help. This avoids spending a search (and its cost) on prompts that do not need one.

The rules engine, at a high level:

- **Skips** searching for prompts that do not benefit, such as pure text transforms, creative writing, stable general knowledge, and code or math questions.
- **Forces** a search when the prompt shows an explicit need for fresh information.
- **Falls back to an LLM router** only when the request is genuinely ambiguous and `mode` is `auto`. The router makes the call in those in-between cases.

So `mode: on` leans toward searching, `mode: off` disables it, and `mode: auto` hands the decision to the rules engine plus the router. If a search is skipped, you are not charged for one, and the response metadata reflects that nothing was retrieved.

## Provider Routing

Web search requires a model whose provider can serve it — today an **OpenRouter, xAI, NEAR, or Phala** model. Enabling it on a model outside that set is rejected with `web_search requires an OpenRouter, xAI, NEAR, or Phala model`, not silently dropped. You can confirm per model with the web-search capability in `GET /v1/models`.

## What the Pipeline is

The pipeline is **gateway-native**, backed by **Brave** as the search backend. "Gateway-native" means Nito owns the search-and-retrieve step rather than blindly forwarding a provider's hosted search: it runs the rules engine, queries the backend, retrieves and trims sources to your `max_sources` and `max_context_tokens` budgets, and injects the result into the model's context. Outbound queries are redacted to strip secrets and PII before they leave the gateway.

## Worked Example

A request that asks for fresh information and localizes results:

```bash wrap
curl https://nito.dev.zprotocol.org/v1/chat/completions \
  -H "Authorization: Bearer $NITO_API_KEY" \
  -H "Content-Type: application/json" \
  -d '{
    "model": "x-ai/grok-4.5",
    "messages": [
      { "role": "user", "content": "What are the top local news stories today?" }
    ],
    "web_search": {
      "mode": "auto",
      "max_sources": 5,
      "max_context_tokens": 4000,
      "location": { "city": "Austin", "state": "TX", "country": "US" }
    }
  }'
```

The response is an ordinary chat completion with two additions:

- A top-level **`web_search`** metadata object describing the search the gateway performed and the sources it used.
- Search counters under **`usage.server_tool_use`**: `web_search_requests` and `web_search_results`.

```json wrap
{
  "id": "...",
  "object": "chat.completion",
  "model": "x-ai/grok-4.5",
  "choices": [ { "index": 0, "message": { "role": "assistant", "content": "..." }, "finish_reason": "stop" } ],
  "web_search": { "...": "search metadata and sources" },
  "usage": {
    "prompt_tokens": 0,
    "completion_tokens": 0,
    "total_tokens": 0,
    "server_tool_use": { "web_search_requests": 1, "web_search_results": 5 }
  }
}
```

## Billing

Web search is billed in addition to the model's own token usage. The search step is metered as pseudo-models: **`brave/llm_context`** and **`brave/web_search`**. These appear alongside your model charges. The `usage.server_tool_use` counters on the response are your handle for reconciling search activity against the bill.
