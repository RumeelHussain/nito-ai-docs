---
title: "Prompt Caching"
description: "Pass provider prompt-cache directives through Nito to cut the cost of long, repeated prompt prefixes. The gateway forwards the hints; the provider does the caching."
---

If you send the same long preamble on call after call, a system prompt, a big set of instructions, a document the model keeps re-reading, you are paying to process those tokens every single time. Prompt caching lets a provider remember that prefix so the next request with the same prefix skips most of the work and bills you at a reduced rate for the cached part. Nito supports this as a **passthrough**: you set the cache directives on your request and the gateway forwards them to the provider that serves your model. This page explains exactly what Nito does and, just as importantly, what it does not do.

If you have not read [Chat Completions](/features/chat-completions), skim it first. Prompt caching is a set of optional fields on that same request.

## What "passthrough" Means

The caching itself happens at the provider, not in Nito. The gateway does **not** store your prompt prefixes and does not maintain its own cache of prefixes. What it does is carry your cache directives through to a provider that supports them (an Anthropic-style ephemeral-breakpoint model), and then surface the provider's cache accounting back to you in the response `usage`.

So the mental model is: you mark which part of the request is the stable, reusable prefix; the provider caches it on its side; subsequent requests that repeat that prefix read from the provider's cache instead of reprocessing it. Nito is the conduit and the meter, not the cache.

One consequence: caching only works on models whose backend supports it. If you set a cache field on a model that does not support caching, the request is rejected with a `400` rather than silently ignored.

## The Request Fields

There are two top-level fields, plus per-part and per-tool variants for finer control.

| **Field** | **Type** | **Notes** |
| --- | --- | --- |
| `prompt_cache_key` | string | A key that identifies a reusable prompt prefix. Maximum **256 bytes**; longer is rejected with `400`. The model must support caching or the call is rejected. |
| `cache_control` | object | A JSON object of provider cache directives (Anthropic-style ephemeral breakpoints), forwarded as-is. The model must support caching. |

Both are optional. Send them only when you actually have a stable prefix worth caching.

### Per-Content-Part and Per-Tool `cache_control`

Beyond the top-level fields, you can attach `cache_control` to finer-grained pieces of the request so the provider knows precisely where the cacheable boundary sits:

- **Per content part.** A `text` content part can carry its own `cache_control`, marking that part as a cache breakpoint. This is how you cache, say, a long document at the start of a user message while leaving the trailing question uncached.
- **Per tool.** A tool declaration can carry its own `cache_control`, so a large, stable set of tool definitions can be cached across calls.

These are passed straight through to the provider. Nito does not interpret the directive contents; it validates that the model supports caching and forwards the rest.

## What it Costs and Saves

Cache accounting comes back in the response `usage` object so you can see exactly what was cached and what was billed at full rate:

- `cache_read_input_tokens` tokens served from the provider's cache (the cheap path).
- `cache_write_input_tokens` tokens written into the cache on this call (a one-time cost to populate it).
- `prompt_tokens_details.cached_tokens` the cached portion of the prompt, reported under the prompt-token detail breakdown. A related `cache_creation_input_tokens` field may also appear there.

The economics are simple. The first call that establishes a prefix pays a small write cost. Every later call that hits the same prefix reads it back at a reduced rate instead of paying full input-token price for those tokens. The longer and more repeated the prefix, the bigger the win.

## When Caching Actually Helps

Caching pays off when a large, identical prefix repeats across many requests. Good fits:

- A long, fixed **system prompt** or instruction block reused on every call.
- A **document or knowledge base** you keep prepending so the model can answer questions about it.
- A large, stable set of **tool definitions** sent on every turn of an agent loop.

It does **not** help when each request is short, or when the prefix changes every time (a cache that is never hit only ever pays the write cost). Put the stable content first and the variable content (the user's actual question) last, so the cacheable boundary is as long as possible.

## Response Caching is a Different Thing, and Nito Does Not Do It

It is easy to confuse prompt caching with **response caching**, returning a previously stored full response when an identical request comes in again. Nito does **not** implement response caching. There is no stored-response lookup; every request runs the model. The only caching Nito participates in is the prompt-prefix passthrough described above. If you need to avoid re-running an identical request, that is a concern for your own application layer, not the gateway.

## Example

The model below is illustrative. Fetch `GET /v1/models` for the live catalog and select a caching-capable model by capability at runtime, because models are added and retired over time.

```bash wrap
curl https://nito.dev.zprotocol.org/v1/chat/completions \
  -H "Authorization: Bearer $NITO_API_KEY" \
  -H "Content-Type: application/json" \
  -d '{
    "model": "x-ai/grok-4.5",
    "prompt_cache_key": "support-assistant-v3",
    "messages": [
      {
        "role": "system",
        "content": [
          {
            "type": "text",
            "text": "You are a support assistant. <long, stable instructions and policy text here>",
            "cache_control": {"type": "ephemeral"}
          }
        ]
      },
      {"role": "user", "content": "How do I reset my password?"}
    ]
  }'
```

On the first call the stable system block is written to the provider's cache (`cache_write_input_tokens`). On later calls that reuse the same `prompt_cache_key` and the same prefix, that block is read back cheaply (`cache_read_input_tokens`), and only the new user question is billed at full input rate.
