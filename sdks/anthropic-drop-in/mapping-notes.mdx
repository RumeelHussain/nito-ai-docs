Nito now serves an Anthropic Messages-compatible endpoint (`/v1/messages` and `/v1/messages/count_tokens`), so Anthropic-shaped code can target Nito directly. For teams that prefer the OpenAI shape, Anthropic-shaped code can also reach Nito through the OpenAI-compatible Chat Completions surface. This page maps the concepts you know from the Anthropic Messages API onto the Chat Completions shape, so that translation is mechanical when you choose it.

<Note>
  Nito serves an Anthropic Messages-compatible endpoint at `POST /v1/messages`, with token counting at `POST /v1/messages/count_tokens`. The mappings below are for teams that prefer to route Anthropic-shaped code through `/v1/chat/completions` instead; both paths are available. Explore them in the [API Playground](/api-playground).

## Field-by-field mapping

| Anthropic Messages concept | Chat Completions equivalent on Nito |
| --- | --- |
| Top-level `system` string | A message with `role: "system"` at the start of `messages` |
| `messages` with `user` / `assistant` roles | `messages` with the same `user` / `assistant` roles |
| `max_tokens` (required by Anthropic) | `max_tokens` (optional here; passed through) |
| `temperature`, `top_p` | `temperature`, `top_p` (passed through) |
| `stop_sequences` | Not part of the documented Chat Completions fields on Nito; do not assume it is read |
| `model` (an Anthropic name) | A Nito catalog ID from `GET /v1/models` |
| `stream: true` | `stream: true` (Server-Sent Events) |
| Response `content` blocks | `choices[0].message.content` (a string) |
| `stop_reason` | `finish_reason` (`stop`, `length`, `tool_calls`, `error`) |
| `usage.input_tokens` / `output_tokens` | `usage.prompt_tokens` / `completion_tokens` |

## System prompts

Anthropic takes a top-level `system` string. The Chat Completions shape has no top-level `system` field; instead, put the system text in a message with `role: "system"` at the start of the array.

```python
# Anthropic-style (top-level system) becomes:
messages=[
    {"role": "system", "content": "You are a concise assistant."},
    {"role": "user", "content": "Say hello."},
]
```

A leftover top-level `system` field is not read on the Chat Completions surface, so convert it rather than leaving it in place.

## Roles

Nito's inference layer accepts `system`, `user`, `assistant`, `tool`, and `developer` roles. The `developer` role is accepted and passed through with no special handling. For an Anthropic codebase, the common path is just `system`, `user`, and `assistant`, with `tool` messages carrying tool results.

## Model names

Anthropic model names are not Nito IDs. Replace them with a Nito catalog ID. There is no static translation table baked into the gateway; pick by capability from `GET /v1/models`.

## Tool use

Anthropic tool use maps to Chat Completions tool calling. Declare function tools in `tools`, the model returns `tool_calls`, your code runs them, and you return the result in a `tool` message paired with the `tool_call_id`. The gateway routes the conversation but does not execute your function tools. See [Tool calling](/features/tool-calling).

## Privacy

Privacy is selected by the model ID, the same as everywhere on Nito. Add `:confidential` or `:encrypted` to raise the tier; only those two suffixes are valid. `:encrypted` is non-streaming and needs extra headers. See [Per-call privacy modes](/privacy/per-call-privacy-modes).

## Things that differ

- **`system_fingerprint` is always `null`** on the response. Do not branch on it.
- **The error envelope has no `type` field.** Errors are `{"error": {"message", "code", "param"}}`; the family is in `code`.
- **There is always exactly one choice** in the response.
- **Token counting** on the Chat Completions surface comes from `usage` on the response. If you need a dedicated count-tokens call, Nito serves the Anthropic Messages-compatible `/v1/messages/count_tokens` endpoint.
