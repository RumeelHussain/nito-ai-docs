If your codebase is built around the Anthropic SDK, this page shows how to reach Nito with the least change, and is honest about where the boundary is today.

<Warning>
The native Anthropic Messages endpoint (`/v1/messages`) and token counting (`/v1/messages/count_tokens`) are **not served** by the gateway. Requests to them return `404`. This means a native Anthropic SDK call such as `client.messages.create(...)` will not reach Nito today. Until the Messages API ships, route Anthropic-shaped code through Nito's OpenAI-compatible Chat Completions surface, as shown below.
</Warning>

## What this means in practice

Nito's compatible surface is the OpenAI Chat Completions shape, not the Anthropic Messages shape. So the "drop-in" for an Anthropic codebase is not "swap the base URL on the Anthropic client and keep going." The native Messages calls would `404`. Instead, you point an OpenAI-style client at Nito and adapt the small set of shape differences (covered in the [mapping notes](/sdks/anthropic-drop-in/mapping-notes)).

Most Anthropic-based applications keep a thin abstraction over the model call, so this change is usually confined to one client-setup file.

## The recommended path today

Use the OpenAI SDK pointed at Nito, and use the Chat Completions message shape.

<CodeGroup>

```python Python
from openai import OpenAI

# Route Anthropic-style code through Nito's OpenAI-compatible surface.
client = OpenAI(
    api_key="z_sk_your_key_here",
    base_url="https://YOUR-NITO-BASE-URL/v1",
)

resp = client.chat.completions.create(
    model="grok-4.3",   # a Nito catalog ID from GET /v1/models
    messages=[
        {"role": "system", "content": "You are a concise assistant."},
        {"role": "user", "content": "Say hello in one short sentence."},
    ],
)

print(resp.choices[0].message.content)
```

```typescript TypeScript
import OpenAI from "openai";

const client = new OpenAI({
  apiKey: "z_sk_your_key_here",
  baseURL: "https://YOUR-NITO-BASE-URL/v1",
});

const resp = await client.chat.completions.create({
  model: "grok-4.3",
  messages: [
    { role: "system", content: "You are a concise assistant." },
    { role: "user", content: "Say hello in one short sentence." },
  ],
});

console.log(resp.choices[0].message.content);
```

## Can I keep the Anthropic SDK?

Not against the gateway today. The Anthropic SDK's primary call is `messages.create`, which targets `/v1/messages`. That path returns `404` on the gateway, and so does `/v1/messages/count_tokens`. There is no shim in the gateway that accepts the Messages shape. The path that works is the OpenAI-compatible client shown above. This page will be updated if and when a Messages-shaped surface ships.

## Privacy

As everywhere on Nito, privacy rides on the model ID. Add `:tee` or `:e2e-tee` to the model string to raise the tier, with no other code change. Only those two suffixes are valid. `:e2e-tee` is non-streaming and needs extra headers. See [Per-call privacy modes](/privacy/per-call-privacy-modes).

## Common mistakes

- **Calling `/v1/messages`.** It returns `404` today. Use `/v1/chat/completions`.
- **Keeping the native Anthropic client.** `client.messages.create(...)` will not reach Nito. Use an OpenAI-style client against `/v1`.
- **Carrying over an Anthropic model name.** Replace it with a Nito catalog ID.
- **Keeping a top-level `system` string.** Move it into the messages array; see the [mapping notes](/sdks/anthropic-drop-in/mapping-notes).
