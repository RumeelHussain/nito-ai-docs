If your codebase is built around the Anthropic SDK, this page shows how to reach Nito with the least change.

<Note>
  Nito now serves an Anthropic Messages-compatible endpoint: `POST /v1/messages` and token counting at `POST /v1/messages/count_tokens`. The gateway validates the Anthropic Messages shape and maps it to Nito's canonical chat contracts, so a native Anthropic SDK call such as `client.messages.create(...)` can target Nito. Nito's OpenAI-compatible Chat Completions surface remains a valid alternative for teams that prefer the OpenAI shape, as shown below. Explore both in the [API Playground](/api-playground).

## What this means in practice

You have two working paths into Nito from an Anthropic codebase. You can point the Anthropic SDK at Nito's Anthropic Messages-compatible endpoint (`/v1/messages`), or you can point an OpenAI-style client at Nito's Chat Completions surface and adapt the small set of shape differences (covered in the [mapping notes](/sdks/anthropic-drop-in/mapping-notes)). The OpenAI path is an alternative for teams that already standardize on the OpenAI shape.

Most Anthropic-based applications keep a thin abstraction over the model call, so either choice is usually confined to one client-setup file.

## Routing through Chat Completions

For teams that prefer the OpenAI shape, use the OpenAI SDK pointed at Nito, and use the Chat Completions message shape.

<CodeGroup>

```python Python
from openai import OpenAI

# Route Anthropic-style code through Nito's OpenAI-compatible surface.
client = OpenAI(
    api_key="z_sk_your_key_here",
    base_url="https://nito.dev.zprotocol.org/v1",
)

resp = client.chat.completions.create(
    model="x-ai/grok-4.5",   # a Nito catalog ID from GET /v1/models
    messages=[
        {"role": "system", "content": "You are a concise assistant."},
        {"role": "user", "content": "Say hello in one short sentence."},
    ],
)

print(resp.choices[0].message.content)
```

```typescript TypeScript
import OpenAI from "openai";

const client = new OpenAI({
  apiKey: "z_sk_your_key_here",
  baseURL: "https://nito.dev.zprotocol.org/v1",
});

const resp = await client.chat.completions.create({
  model: "x-ai/grok-4.5",
  messages: [
    { role: "system", content: "You are a concise assistant." },
    { role: "user", content: "Say hello in one short sentence." },
  ],
});

console.log(resp.choices[0].message.content);
```

## Can I keep the Anthropic SDK?

Yes. The Anthropic SDK's primary call is `messages.create`, which targets `/v1/messages`. Nito now serves an Anthropic Messages-compatible endpoint at that path, along with `/v1/messages/count_tokens`, so the gateway accepts the Messages shape and maps it to Nito's canonical contracts. The OpenAI-compatible client shown above remains a valid alternative. Try both against the [API Playground](/api-playground).

## Privacy

As everywhere on Nito, privacy rides on the model ID. Add `:confidential` or `:encrypted` to the model string to raise the tier, with no other code change. Only those two suffixes are valid. `:encrypted` is non-streaming and needs extra headers. See [Per-call privacy modes](/privacy/per-call-privacy-modes).

## Common mistakes

- **Carrying over an Anthropic model name.** Replace it with a Nito catalog ID, whichever path you use.
- **Keeping a top-level `system` string.** Move it into the messages array; see the [mapping notes](/sdks/anthropic-drop-in/mapping-notes).
