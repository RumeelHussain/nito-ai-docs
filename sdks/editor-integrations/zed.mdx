Zed's assistant supports custom OpenAI-compatible providers. Because Nito exposes an OpenAI-compatible API, you can route Zed's assistant through Nito by filling in that custom-endpoint configuration. You are not selecting a named "Nito" provider. You are using Zed's OpenAI-compatible provider option and pointing it at Nito.

## Configuration

Zed has a first-class `openai_compatible` provider. Add it to `settings.json` under `language_models`, listing the Nito models you want (Zed does not auto-discover them):

```json
{
  "language_models": {
    "openai_compatible": {
      "nito": {
        "api_url": "https://nito.dev.zprotocol.org/v1",
        "available_models": [
          { "name": "x-ai/grok-4.5", "display_name": "Grok 4.5", "max_tokens": 128000 }
        ]
      }
    }
  }
}
```

- **`api_url`** must end in `/v1`.
- **The API key goes in the UI, not the file** — add it under Agent Settings → LLM Providers (Zed stores it in the keychain). Do not put `z_sk_...` in `settings.json`.
- Each entry in `available_models` needs a `name` (the Nito catalog ID). Add more entries for more models.

## Finding a model ID

Zed will not auto-discover Nito's catalog, so list available models in the custom provider's configuration. Fetch the live list and use one of the returned IDs as the model name:

```bash
curl https://nito.dev.zprotocol.org/v1/models \
  -H "Authorization: Bearer z_sk_your_key_here"
```

Register the Nito ID as a model name in Zed's provider configuration so it appears in the model selector.

## Choosing a privacy tier

Privacy is selected by the model ID, so no separate Zed setting is required. Append `:confidential` for a TEE-backed run with a verifiable attestation, for example `openai/gpt-oss-120b:confidential`. Only `:confidential` and `:encrypted` are valid suffixes, and the model must support the tier. `:encrypted` does not stream and needs extra headers, so it is generally not usable from Zed's assistant. Use `GET /v1/models` to find attestation-capable models, and see [Per-call privacy modes](/privacy/per-call-privacy-modes).

## What to expect

- Zed sends standard Chat Completions requests, which Nito serves at `POST /v1/chat/completions`. Streaming and tool calls behave as they do with the OpenAI SDK.
- Nito now also serves an OpenAI Responses-compatible endpoint (`/v1/responses`) and an Anthropic Messages-compatible endpoint (`/v1/messages`), but this guide keeps Zed on the Chat Completions path for the widest support.
- If a model is rejected, confirm you used a Nito catalog ID and that the base URL ends in `/v1`.

## Common mistakes

- **Leaving an OpenAI model name in.** Replace `gpt-4o` with a Nito catalog ID from `GET /v1/models`.
- **Dropping `/v1` from the base URL.** The Chat Completions route will not be reached without it.
- **Assuming a built-in Nito provider.** There is none. Use Zed's OpenAI-compatible custom provider.
