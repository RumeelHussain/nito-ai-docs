You do not need an SDK to call Nito. Because the gateway is OpenAI-compatible, an HTTP client like [Postman](https://www.postman.com/) or [Bruno](https://www.usebruno.com/) is enough to send requests and inspect raw responses. This is the fastest way to verify your key, explore the catalog, and debug request shapes.

There is no ready-made collection file in this repository yet, so this page shows how to build the requests by hand.

## Setup

Configure these once, ideally as environment variables in Postman or Bruno so you can switch them without editing each request.

| Setting | Value |
| --- | --- |
| Base URL | `https://YOUR-NITO-BASE-URL/v1` |
| Auth header | `Authorization: Bearer z_sk_your_key_here` |
| Content type | `Content-Type: application/json` |

> **TODO (needs SME):** the canonical production base URL. Replace `https://YOUR-NITO-BASE-URL` with the real gateway URL.

Use a Nito `z_sk_` key, not an OpenAI `sk-` key. Store the key as a secret variable so it does not get committed or shared.

## List the catalog first

Before sending a chat request, fetch the live model catalog so you use a valid ID:

```text
GET https://YOUR-NITO-BASE-URL/v1/models
Authorization: Bearer z_sk_your_key_here
```

The response lists the model IDs you can pass as `model`. The example `grok-4.3` below is illustrative; pick a real ID from this response, because the catalog changes over time.

## Sample chat completion request

```text
POST https://YOUR-NITO-BASE-URL/v1/chat/completions
Authorization: Bearer z_sk_your_key_here
Content-Type: application/json
```

Body:

```json
{
  "model": "grok-4.3",
  "messages": [
    { "role": "system", "content": "You are concise." },
    { "role": "user", "content": "Say hello in one short sentence." }
  ]
}
```

A successful response is a standard `chat.completion` object. The assistant's text is at `choices[0].message.content`, and `usage` reports the tokens billed. There is always exactly one choice, and `system_fingerprint` is always `null`.

## Selecting privacy

Privacy rides on the model ID. To run inside an attested environment, change one field:

```json
{ "model": "grok-4.3:tee" }
```

Only `:tee` and `:e2e-tee` are valid suffixes, and the model must support that tier. Note that `:e2e-tee` is non-streaming and needs extra headers, so it is not well suited to a simple Postman or Bruno request. See [Per-call privacy modes](/privacy/per-call-privacy-modes).

## Other OpenAI-compatible surfaces

The same base URL and bearer auth reach the other compatible endpoints:

| Method | Path | Purpose |
| --- | --- | --- |
| POST | `/v1/chat/completions` | Chat and text |
| POST | `/v1/embeddings` | Vectors |
| POST | `/v1/images/generations` | Text-to-image |
| GET | `/v1/models` | List the catalog |

The OpenAI Responses API (`/v1/responses`) is served as an OpenAI Responses-compatible endpoint, and the Anthropic Messages API (`/v1/messages`, `/v1/messages/count_tokens`) is served as an Anthropic Messages-compatible endpoint. Chat Completions remains the primary documented surface; explore the others in the [API Playground](/api-playground).

## Reading errors

Errors come back as `{"error": {"message", "code", "param"}}`. There is no `type` field; the error family is in `code`. On a `429`, look for the `Retry-After` header. All responses carry `Cache-Control: no-store`.

## Streaming in an HTTP client

To stream, add `"stream": true` to the body. The response is Server-Sent Events: lines prefixed `data: ` with JSON chunks, ending with `data: [DONE]`. Postman and Bruno can display the streamed body, though a terminal `curl` is often easier to read for streaming. See [Streaming](/features/streaming).
