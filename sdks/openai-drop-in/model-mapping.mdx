The one thing that does not carry over from an OpenAI codebase is the model name. `gpt-4o`, `gpt-4o-mini`, and similar strings are OpenAI catalog names. They are **<u>not</u>** Nito IDs, and sending one names a model the gateway does not have. This page explains the principle and how to choose the right model ID.

## The Principle for Model Names

Nito's catalog is **runtime-discovered**, not a fixed list baked into the gateway. Models are synced from provider APIs into the catalog, so the set of available models changes over time as models are added and retired. There is no hardcoded model table inside the gateway, and therefore no static, frozen list to copy into your code.

The practical consequence: **_do not hard-code a model string and assume it will always exist_**. Fetch the live catalog and select by capability.

## Find the Right Model ID with `GET /v1/models`

The authoritative source of valid model IDs is the catalog endpoint. With the OpenAI SDK pointed at Nito:

<CodeGroup>

```python Python lines wrap
from openai import OpenAI

client = OpenAI(
    api_key="z_sk_your_key_here",
    base_url="https://nito.dev.zprotocol.org/v1",
)

for m in client.models.list().data:
    print(m.id)
```

```typescript TypeScript lines wrap
import OpenAI from "openai";

const client = new OpenAI({
  apiKey: "z_sk_your_key_here",
  baseURL: "https://nito.dev.zprotocol.org/v1",
});

const models = await client.models.list();
for (const m of models.data) {
  console.log(m.id);
}
```

## How a Nito Model ID is Built

A Nito public model ID is a canonical model name with an optional privacy suffix:

```text
<canonical-model-id>[:confidential | :encrypted]
```

- The base is the canonical model ID from the catalog, e.g. `x-ai/grok-4.5` or `openai/gpt-oss-120b`.
- A `:confidential` or `:encrypted` suffix selects a stronger privacy mode. Only those two suffixes are valid; any other (`:standard`, `:zdr`, `:anon`, `:private`, …) is rejected.

Privacy variants are **published as their own catalog entries**, only for the models that support them — not every model has one. For example, `x-ai/grok-4.5` has no confidential variant, while `openai/gpt-oss-120b:confidential` and `openai/gpt-oss-120b:encrypted` are listed as distinct IDs. So the reliable way to select a private tier is to call `GET /v1/models` and pick an ID that already ends in `:confidential` or `:encrypted`, rather than appending a suffix to an arbitrary model. The privacy decision rides entirely on the model ID; there is no separate tier parameter. See [Per-call privacy modes](/privacy/per-call-privacy-modes).

## Mapping OpenAI Names to Nito Model IDs

There is no mapping table inside the gateway that translates an OpenAI name to a Nito ID, so this documentation does not invent one. The correct move is always to look up an equivalent in the live catalog by the capability you need (chat, embeddings, image generation, reasoning, attestation support) rather than by matching brand names.

## A Migration Checklist

1. List the OpenAI model names your code uses today.
2. Call `GET /v1/models` against Nito and note the IDs that match each capability.
3. Replace each hard-coded OpenAI name with the chosen Nito ID, or better, select from the catalog at runtime.
4. If a call needs stronger privacy, append `:confidential` or `:encrypted` and confirm the model supports that tier.
