You do not need a Nito-specific SDK to call Nito. The gateway is OpenAI-compatible on its main surfaces, so the official OpenAI SDK works as a drop-in: change the base URL, change the API key, and use a Nito catalog model ID. Codebases built around the Anthropic SDK can reach Nito too, with one honest caveat covered below.

This section is the deep version of the two quickstarts. If you just want the fastest path, start with the [OpenAI SDK quickstart](/get-started/quickstart-openai-sdk) or the [Anthropic SDK quickstart](/get-started/quickstart-anthropic-sdk), then come back here when you need the compatibility details.

## The compatible surface

Nito speaks the OpenAI shape on four surfaces:

- `POST /v1/chat/completions` the primary chat and text surface.
- `POST /v1/embeddings` vectors for search and retrieval.
- `POST /v1/images/generations` text-to-image.
- `GET /v1/models` the live catalog.

"Drop-in" means exactly this: set the SDK's `base_url` to your Nito base plus `/v1`, set the API key to a `z_sk_` key, and name a Nito catalog model ID. The request and response shapes are the ones the OpenAI SDK already expects.

Nito also serves compatible endpoints for the broader OpenAI and Anthropic worlds:

- The OpenAI **Responses API** (`/v1/responses`) as an OpenAI Responses-compatible endpoint. Chat Completions remains the primary documented text surface.
- The Anthropic **Messages API** (`/v1/messages`) and token counting (`/v1/messages/count_tokens`) as Anthropic Messages-compatible endpoints. See the [Anthropic drop-in pages](/sdks/anthropic-drop-in/base-url-swap) for what this means in practice.

## Language matrix

The supported path today is the official OpenAI SDK (and, with the caveat below, the Anthropic SDK) in Python and TypeScript. Both languages call the same gateway surfaces with the same request shapes.

| Language | SDK | Status | Surface used |
| --- | --- | --- | --- |
| Python | `openai` | Supported drop-in | `/v1/chat/completions`, `/v1/embeddings`, `/v1/images/generations`, `/v1/models` |
| TypeScript | `openai` | Supported drop-in | same as above |
| Python | `anthropic` (Messages) | Served (Anthropic Messages-compatible) | native `/v1/messages`, or route through Chat Completions |
| TypeScript | `@anthropic-ai/sdk` (Messages) | Served (Anthropic Messages-compatible) | same as above |

For the Anthropic SDKs you can target Nito's Anthropic Messages-compatible `/v1/messages` endpoint directly, or route Anthropic-shaped code through the OpenAI-compatible Chat Completions surface. Both paths are covered on the Anthropic pages.

## Documented separately (deferred this pass)

Nito also has first-party tooling that lives in its own repositories. These are real, but their APIs are out of scope for this documentation pass and are intentionally not described here. Do not build against an interface inferred from this page.

- **Native Nito SDK** deferred, documented separately.
- **Agent SDK** deferred, documented separately.
- **CLI (`z`)** deferred, documented separately.
- **MCP server** deferred, documented separately. The gateway exposes an MCP transport at `/mcp` (Streamable HTTP) with OAuth `z_mcp_` tokens and the `mcp:inference` scope, but the MCP server and its protocol are documented in their own repo.

When those pages exist, they will be linked from here. Until then, treat the OpenAI and Anthropic drop-in paths as the documented way to integrate.

## Privacy, briefly

Across every SDK path, privacy is selected by the model ID, not by a separate parameter. Add `:confidential` or `:encrypted` to the model string to raise the tier; no other code changes. `system_fingerprint` is always `null`, and the error envelope carries the error family in `code` with no `type` field. Streaming, tool calling, prompt caching, and reasoning are supported on the Chat Completions surface. See [Per-call privacy modes](/privacy/per-call-privacy-modes).
