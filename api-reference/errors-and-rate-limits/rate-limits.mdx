Nito enforces rate limits from a **policy table** of limiter groups, each with limits per **scope** (global, IP, org, API key, OAuth token, session, and more). The defaults below are the current published values; they are **deployment-configurable**, so a given environment may differ.

<Note>
Nito emits **no `X-RateLimit-*` headers** (no `limit`, `remaining`, or `reset`). The only signal is a `Retry-After` header, and even that is **not always present** (see [Signalling](#what-a-429-looks-like)). Pace your clients with your own accounting and backoff rather than reading a remaining-quota header — none is sent.
</Note>

<Accordion title="How scopes and per-key limits work">
A limit applies per **scope** — global, ip, org, `api_key`, `oauth_token`, session, user, the `mcp_*` scopes, subject, and command. A request must pass **every** scope that applies to it; the tightest one wins.

**Per-key values are global group defaults, not stored per individual key.** Two API keys in the same group share the same default rate; the policy table holds one default per scope, not a custom number per key. Changing the rate for a specific workload is a deployment-level policy change.
</Accordion>

## Concurrency (in-flight) limits

`inference_generate` also bounds **concurrent in-flight requests**, independent of the per-minute rate: **25 per org** and **10** per api_key / oauth_token / session / mcp_token / user. Exceeding the concurrency bound is rejected with a `429` even when you are under the per-minute rate. If you fan out many long-running generations at once, this is the bound you'll hit first.

## What a 429 looks like

Signalling depends on **why** you were limited:

- **Request-rate `429`** (per-minute limit) — includes a `Retry-After` header with the seconds to wait, when the limiter can compute a wait duration.
- **In-flight concurrency `429`** — **may not include `Retry-After`** (there is no fixed window to wait out; retry after a request frees up).
- **Quota `429`** (`insufficient_quota`) — **may not include `Retry-After`** (see [Quota is not a rate limit](#quota-is-not-a-rate-limit)).

There are **no `X-RateLimit-*` headers** in any case.

```http
HTTP/1.1 429 Too Many Requests
Retry-After: 60
Content-Type: application/json; charset=utf-8

{"error":{"message":"rate limited","code":"rate_limited"}}
```

### Error shapes by surface

The body shape depends on which surface you called:

| Surface | Shape |
|:--------|:------|
| Public API (JSON) | OpenAI-style envelope: `{"error":{"code":"rate_limited","message":"…"}}` |
| Browser JSON routes | Flat: `{"error":"…","code":"rate_limited"}` |
| OAuth routes | OAuth-style: `{"error":"rate_limited"}` |
| MCP transport | May be **plain text** |
| MCP tools | Surfaced as a **tool error** |

## Quota is not a rate limit

`insufficient_quota` is a **billing/quota** decision, not a Redis rate-limit decision — a different system with a different meaning, even though both return `429`.

- **Free-tier prompt quotas** are counted per **UTC calendar day**, currently defaulting to **10 text prompts/day** and **15 image prompts/day**. Exhausting them returns `429` with `code: "insufficient_quota"`.
- **Paid entitlement failures** (e.g. no balance) can return **`402 payment_required`** rather than `429`.

So distinguish the two: `rate_limited` is transient (retry after the wait); `insufficient_quota` / `402` is a spend or usage cap that needs a top-up or a new day, not a retry.

## Fail-open versus fail-closed

If the limiter's own backend (Redis) errors, behavior depends on the group:

<Accordion title="Fail-closed groups (reject when the limiter is unavailable)">
These groups **fail closed** — if the limiter can't be consulted, the request is rejected rather than let through unbounded:

- `auth_interactive`
- `inference_generate`
- `inference_count`
- `mcp_transport`
- `mcp_tool_inference`
- `mcp_tool_metadata`
- `oauth`
- `org_admin`
- `spa_api_account_org`

All other groups (including **`model_list`**) **fail open** — the request proceeds on a limiter-backend fault.
</Accordion>

A Redis ping failure triggers reconnecting behavior; fail-closed groups keep rejecting while the limiter is unavailable.

## Configuration

- Runtime policy overrides are loaded from the **`RATE_LIMIT_POLICIES_FILE`** environment variable (a path to a policy file). **Inline JSON policy config is intentionally unsupported.**
- A bad policy or config is **fatal at startup** — the gateway will not boot with an invalid policy.

## Handling limits in your client

- Treat `rate_limited` `429`s as retryable, honoring `Retry-After` when present; otherwise back off ([Retry behavior](/api-reference/errors-and-rate-limits/retry-behavior)).
- Do not poll for a remaining-quota header — it does not exist.
- Bound your own concurrency to stay under the in-flight limits (25 per org, 10 per key on `inference_generate`).
- Treat `insufficient_quota` and `402` as **not** retryable in a tight loop — they need a top-up or a new UTC day, not immediate retry.
