Nito enforces rate limits from a **policy table** of limiter groups, each with per-scope limits. The defaults below are a useful reference; values are **deployment-configurable**, so a given production environment may differ.

<Note>
On a `429`, Nito sends **only** a `Retry-After` header (seconds to wait). There are **no `X-RateLimit-*` headers** — no `limit`, `remaining`, or `reset`. Read the wait time from `Retry-After` and back off; don't build client logic around a remaining-quota or reset header, because none is sent.
</Note>

## Default limits

### `inference_generate`

Chat, fusion, image, and embeddings generation.

| Scope | Limit |
|:------|------------------:|
| org | 600 / min |
| api_key | 120 / min |
| session | 120 / min |
| org (in-flight) | 25 concurrent |
| api_key (in-flight) | 10 concurrent |

### Other groups

| Group | Scope | Limit |
|:------|:------|------------:|
| `inference_count` | api_key | 300 / min |
| `model_list` | api_key | 600 / min |

<Accordion title="How scopes and per-key limits work">
The policy table defines limits per **scope** — global, ip, org, `api_key`, `oauth_token`, session, user, `oauth_client`, the `mcp_*` groups, subject, and command.

**Per-key values are global group defaults, not stored per individual key.** Two API keys in the same group share the same default rate; the table holds one default per scope, not a custom number per key. Changing the rate for a specific workload is a deployment-level policy change, not a per-key setting.
</Accordion>

## Concurrency (in-flight) limits

`inference_generate` also bounds **concurrent in-flight requests**, independent of the per-minute rate:

| Scope | In-flight limit |
|:------|------------------:|
| org | 25 concurrent |
| api_key | 10 concurrent |

Exceeding the concurrency bound is rate-limited the same way (`429` + `Retry-After`) even when you are under the per-minute limit. If you fan out many long-running generations at once, this is the bound you will hit first.

## What you get on a 429

- HTTP status `429`.
- A `Retry-After` header with the seconds to wait.
- The [error envelope](/api-reference/errors-and-rate-limits/error-envelope) with `code` either `rate_limited` (transient — retry after the wait) or `insufficient_quota` (a spend or usage cap — may need a quota change).

```http
HTTP/1.1 429 Too Many Requests
Retry-After: 60
Content-Type: application/json; charset=utf-8

{"error":{"message":"rate limited","code":"rate_limited"}}
```

First-party browser routes return the same `429` and `Retry-After`, but with a flat body (`{"code":"rate_limited","error":"rate limited"}`) rather than the nested `error` envelope.

<Accordion title="Fail-open vs fail-closed groups">
If the limiter's own infrastructure errors (it cannot be consulted), most groups **fail open** — the request proceeds rather than being rejected on an infrastructure fault.

A **fail-closed** set rejects requests when the limiter cannot be consulted, because letting them through unbounded is riskier than a brief outage:

- `auth_interactive`
- `inference_generate`
- `inference_count`
- `mcp_*`
- `oauth`
- `org_admin`
- `spa_api_account_org`

Everything outside this set fails open.
</Accordion>

## Handling limits in your client

- Treat `429` as retryable, but only after the `Retry-After` interval. See [Retry behavior](/api-reference/errors-and-rate-limits/retry-behavior).
- Do not poll for a remaining-quota header — it does not exist.
- Bound your own concurrency to stay under the in-flight limits (25 per org, 10 per key on `inference_generate`).
- Distinguish `rate_limited` (retry after wait) from `insufficient_quota` (spend/usage cap) even though both return `429`.
