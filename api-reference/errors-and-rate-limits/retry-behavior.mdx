---
title: Retry behavior
description: Which Nito errors are safe to retry, which are not, and why live streams are never replayed.
---

Not every error should be retried. Retrying a `400` will just fail again; retrying a `429` after waiting will usually succeed. This page sorts the error families into retryable and non-retryable.

## Retryable

| Status | Codes | How to retry |
|---|---|---|
| `429` | `rate_limited`, `insufficient_quota` | Wait for the `Retry-After` header, then retry. |
| `502` | `provider_error` | Transient upstream failure. Retry with [backoff](/api-reference/errors-and-rate-limits/backoff). |
| `503` | `unavailable` | Transient. Retry with backoff. |

`502` and `503` are treated as transient upstream conditions and are safe to retry with exponential backoff.

## Not retryable

| Status | Meaning |
|---|---|
| `400` | The request is wrong (`invalid_request`, `model_unavailable`). Fix it first. |
| `401` | The credential is wrong (`unauthorized`). Fix the key. |
| `403` | Not permitted (`forbidden`). |

Retrying these unchanged will reproduce the same failure. Change the request, the credential, or the permissions before sending again.

## Streams are never replayed

The gateway never replays a live stream. If a streaming completion fails after bytes have been written, you receive a terminal error chunk (`finish_reason: "error"`); the gateway does not silently restart the generation. To retry, start a new request from the client.
