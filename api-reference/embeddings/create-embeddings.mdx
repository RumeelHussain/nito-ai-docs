Create embedding vectors for one or more input strings.

```
POST /v1/embeddings
POST /api/v1/embeddings
```

Both paths share one handler (`apiEmbeddingsHandler`). The bare `/v1` path is OpenAI-compatible; the `/api/v1` path is the first-party prefix. The request and response are identical.

## Request

| Field | Type | Required | Notes |
|---|---|---|---|
| `input` | string or array of strings | yes | The text to embed. Token-ID arrays are rejected. See [Input limits](/api-reference/embeddings/input-limits) for the count and size caps. |
| `model` | string | no | Falls back to the default embedding model when omitted. See [Embedding models](/api-reference/embeddings/embedding-models). |
| `encoding_format` | string | no | `""` or `"float"` only. `"base64"` is **not** supported and returns `400`. |
| `dimensions` | integer | no | Requested output dimensions, bounded `1`–`3072`. See [Vector dimensions](/api-reference/embeddings/vector-dimensions). |
| `user` | string | no | Opaque caller-supplied identifier, passed through. |

Non-obvious points:

- **Token-ID arrays are rejected.** Unlike some OpenAI-compatible servers, you cannot pass pre-tokenized integer arrays as `input`. Send raw strings.
- **`base64` is not an option.** The only accepted `encoding_format` values are the empty string and `"float"`. Anything else fails with `400`.
- **`dimensions` is a request bound, not a per-model guarantee.** The gateway validates that the value falls in `1`–`3072`, but it does not know each model's native output length; the provider returns the actual vector length at runtime. See [Vector dimensions](/api-reference/embeddings/vector-dimensions).

## Response

```json
{
  "object": "list",
  "model": "Qwen/Qwen3-Embedding-0.6B",
  "data": [
    {
      "object": "embedding",
      "index": 0,
      "embedding": [0.0123, -0.0456, 0.0789]
    }
  ],
  "usage": {
    "prompt_tokens": 8,
    "total_tokens": 8
  }
}
```

| Field | Type | Notes |
|---|---|---|
| `object` | string | Always `"list"`. |
| `model` | string | The model that produced the vectors. |
| `data` | array | One entry per input, in request order. |
| `data[].object` | string | Always `"embedding"`. |
| `data[].index` | integer | Position of this vector in the input list. |
| `data[].embedding` | array of float | The vector. Length is set by the provider at runtime. |
| `usage.prompt_tokens` | integer | Tokens counted from the input. |
| `usage.total_tokens` | integer | Total billable tokens. |

The `data` array preserves input order: `data[i]` is the embedding for `input[i]`. Match results to inputs by `index`, not by assuming a stable iteration order from your client.

## Worked call

```bash
curl "https://nito.dev.zprotocol.org/v1/embeddings" \
  -H "Authorization: Bearer z_sk_your_key_here" \
  -H "Content-Type: application/json" \
  -d '{
    "model": "Qwen/Qwen3-Embedding-0.6B",
    "input": ["the quick brown fox", "lazy dog"]
  }'
```

The example model id `Qwen/Qwen3-Embedding-0.6B` is the native default embedding model. To discover what is actually available in your deployment, call `GET /v1/models?type=embedding`; see [Embedding models](/api-reference/embeddings/embedding-models).

Batching a request with two inputs returns two entries:

```json
{
  "object": "list",
  "model": "Qwen/Qwen3-Embedding-0.6B",
  "data": [
    { "object": "embedding", "index": 0, "embedding": [0.01, -0.02] },
    { "object": "embedding", "index": 1, "embedding": [0.03, -0.04] }
  ],
  "usage": { "prompt_tokens": 9, "total_tokens": 9 }
}
```

## Errors

Validation and decode failures return the canonical error envelope with a `400`. Common causes: a token-ID array in `input`, `encoding_format: "base64"`, a `dimensions` value outside `1`–`3072`, more than 96 inputs, or an input string over 32 KiB. See [Input limits](/api-reference/embeddings/input-limits) and [Error format](/api-reference/overview/error-format).
