title: Vector dimensions
description: The dimensions request bound and why per-model output lengths are not encoded in the gateway.

The embeddings endpoint accepts an optional `dimensions` field that requests a specific output vector length.

## The `dimensions` bound

| Field | Type | Bound |
|---|---|---|
| `dimensions` | integer | `1`–`3072` inclusive |

A `dimensions` value outside `1`–`3072` is rejected with `400`. The bound is validated by the gateway before the request reaches the provider.

```bash
curl "https://nito.dev.zprotocol.org/v1/embeddings" \
  -H "Authorization: Bearer z_sk_your_key_here" \
  -H "Content-Type: application/json" \
  -d '{
    "model": "Qwen/Qwen3-Embedding-0.6B",
    "input": "hello world",
    "dimensions": 1024
  }'
```

## Output length is set at runtime, not by the gateway

The `dimensions` field is a **request bound**, not a per-model guarantee. The gateway does not encode each model's native output dimension. The provider returns the actual vector length at runtime, and that length is whatever appears in `data[].embedding`.

In practice this means:

- A model may or may not honor a requested `dimensions` value. Whether truncation or projection is supported depends on the model, not the gateway.
- Do not assume a fixed vector length from the gateway docs. Read the length from the returned `embedding` array, or measure it once per model.
- If you store vectors, key your storage schema on the length you observe at runtime for the specific model you call.
