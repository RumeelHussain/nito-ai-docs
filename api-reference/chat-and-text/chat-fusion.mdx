Chat fusion asks 2 to 3 models the same prompt concurrently, then runs one synthesis ("judge") pass that merges their answers into a single output. It is **non-streaming only**. This page is the schema reference; for the conceptual model see [Multi-model routing](/features/multi-model-routing).

## Endpoint

```
POST /api/v1/chat/fusion
```

There is **no `/v1` alias** for fusion. The handler is `apiChatFusionHandler`. Each participating model runs under a per-participant budget of 150 seconds.

## Request

| Field | Type | Required | Notes |
|---|---|---|---|
| `messages` | array | Yes | The conversation, same shape as chat completions. See [Message format](/api-reference/chat-and-text/message-format). |
| `models` | array | Yes | 2 to 3 distinct model IDs (`fusionMinModels = 2`, `fusionMaxModels = 3`). |
| `temperature` | number | No | Sampling temperature applied to participants. |
| `top_p` | number | No | Nucleus sampling. |
| `max_tokens` | integer | No | Cap on generated tokens per participant. |
| `user` | string | No | Opaque end-user identifier. |
| `web_search` | bool or object | No | Let participants search the web. |
| `web_fetch` | boolean | No | Let participants fetch a URL. |
| `tools` | array | No | Hosted web tools only. Function tools are rejected. |

### Privacy and the model suffix

Privacy mode is selected the same way as chat completions: through the model-ID suffix on each entry in `models`. Two constraints are specific to fusion:

- You **cannot mix TEE and non-TEE** models in a single fusion request.
- The `:encrypted` suffix is **rejected** for fusion.

## Response

A successful call returns an object with `object` set to `fusion.completion`:

| Field | Type | Notes |
|---|---|---|
| `object` | string | Always `fusion.completion`. |
| `judge` | string | The model used for the synthesis pass. |
| `synthesis` | string | The merged final answer. |
| `synthesis_prompt_tokens` | integer | Prompt tokens consumed by the synthesis pass. |
| `synthesis_tokens` | integer | Tokens generated by the synthesis pass. |
| `synthesis_attestation` | object | Optional. Attestation for the synthesis pass when run under TEE. |
| `models` | array | Per-model results. See below. |
| `free_tier` | object | Optional. Present on free-tier-eligible calls. |

### Per-model result

Each entry in `models` (`apiFusionModelResult`):

| Field | Type | Notes |
|---|---|---|
| `id` | string | Identifier for this participant result. |
| `model` | string | The model that produced this output. |
| `provider` | string | The serving provider. |
| `output` | string | This model's answer. |
| `prompt_tokens` | integer | Prompt tokens for this participant. |
| `tokens` | integer | Tokens generated by this participant. |
| `latency_ms` | integer | Wall-clock latency for this participant. |
| `finish_reason` | string | One of `stop`, `length`, `tool_calls`, `error`. See [Finish reasons](/api-reference/chat-and-text/finish-reasons). |
| `web_search` | object | Optional. Present when this participant ran a web search. |
| `attestation` | object | Optional. Present when this participant ran under TEE. |
| `error` | object | Optional. Present when this participant failed. |

## Partial failure

Fusion returns `200` even on **partial failure**. If one participant fails, its entry carries an `error` and the others still contribute. Inspect each `models[].error` to detect partial degradation rather than relying on the HTTP status alone.
