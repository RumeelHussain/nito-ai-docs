This is the authoritative reference for how Nito prices inference. It covers the billing unit, every rate that can apply to a call, and where to read the live per-model numbers. It does not print a master price table: the per-model figures and the value of one credit are configured live and are not asserted in these docs. See the TODO callouts below for the owning teams.

For the narrative version (holds, settlement, estimating a single call), see [Pricing under Models](/models/pricing). This page is the flat reference.

## The billing unit

Nito bills in **credits**, tracked internally in **nanos** (billionths of a unit) for precision. Every rate and every charge is expressed in nanos. To convert a nanos figure to whole credits, divide by 1,000,000,000 (1e9).

Prices are configured live in the billing system and surfaced on each model's catalog entry, so the authoritative figures always come from the live catalog rather than a static table.

## What gets billed

A single call can accrue charges across several rate categories. Each is priced separately.

| Category | Priced by | Notes |
| --- | --- | --- |
| Input tokens | Per token | Separate rate from output. |
| Output tokens | Per token | Separate rate from input. |
| Cached input tokens | Per token | Cache read and cache write are metered separately from ordinary input. |
| Reasoning output | Per token | Reasoning tokens are billed on top of ordinary output. |
| Server-side tools | Per use | Web search, web fetch, and similar hosted tools bill in addition to model tokens. |
| Image generation | By output | Images are priced by output, not by input tokens. |

Input and output are always separate rates. Cache, reasoning, and server-tool charges are additive on top of the base token cost. Nothing here is free by default; absence of a rate on a model entry means the rate is not yet configured, not that the item is unpriced.

## Where to read per-model cost

Pricing and context window are **not** on the model card by default. When pricing is configured for a model, its `GET /v1/models` entry carries a separate `pricing` object, joined from the billing system at response time. Its fields are expressed in nanos and cover the input and output token rates, cache read and write rates, the reasoning output rate, server-tool rates, an image output rate, and the model's context limit.

Read pricing from the entry so your estimates track the live configuration:

```bash
curl "https://YOUR-NITO-BASE-URL/v1/models" \
  -H "Authorization: Bearer $NITO_API_KEY"
# each entry includes a "pricing" object in nanos when configured
```

## How a call is charged

Before a call, Nito places a hold sized to the expected cost. After the call completes, the hold is settled against actual usage. Failed requests are not billed. The actual charge for a completed call is reported in the response `usage` block (token counts, cache detail, reasoning tokens, and server-tool use) and aggregated in the org usage overview. See [Check usage and credits](/get-started/check-usage-and-credits).

A rough estimate for a text call is:

```text
input_tokens * input_rate
  + output_tokens * output_rate
  + reasoning_tokens * reasoning_rate
  + server_tool_uses * tool_rate
```

all in nanos, with rates pulled from the model's `pricing` object rather than hard-coded.

Server-side tools such as web search and web fetch are billed in addition to the model tokens, at per-use rates that are part of the same live pricing configuration. See [Tool pricing](/api-reference/server-tools/tool-pricing) for the reference.

## Common mistakes

- **Hard-coding prices.** Rates are configured live and can change. Read them from the model entry.
- **Expecting `pricing` on every entry.** It appears only when configured; absence means unset, not free.
- **Treating nanos as whole credits.** Nanos are billionths; divide by 1e9.
- **Forgetting additive costs.** Cache, reasoning tokens, and server tools bill on top of base input and output.
- **Estimating images by token.** Images are priced by output.
