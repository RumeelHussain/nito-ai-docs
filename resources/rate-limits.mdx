This is the canonical rate-limit reference for Nito. It covers how the limiter is structured, the selected published defaults, what you get on a `429`, and how the limiter behaves when its own infrastructure fails. The narrower [rate-limits page in the API reference](/api-reference/errors-and-rate-limits/rate-limits) mirrors these facts scoped to the error-handling flow; this page is the consolidated view.

## How the limiter works

Nito enforces rate limits from a **policy table**. The table defines limiter groups (for example `inference_generate`, `inference_count`, `model_list`) and, within each group, limits per scope. Scopes include global, ip, org, api\_key, oauth\_token, session, user, oauth\_client, the `mcp_*` groups, subject, and command.

The key structural fact: **per-key values are global group defaults, not values stored per individual key.** Two API keys in the same group share the same default rate. The policy table does not hold a custom number for each key; it holds one default that applies to every subject in that scope.

Values are **deployment-configurable** via a policy file, so a given production environment may differ from the defaults listed here.

## Selected defaults

These are the defaults for the listed limiter groups.

### `inference_generate`

Covers chat, fusion, image, and embeddings generation.

| Scope | Limit |
| --- | --- |
| org | 600 / min |
| api\_key | 120 / min |
| session | 120 / min |
| org (in-flight) | 25 concurrent |
| api\_key (in-flight) | 10 concurrent |

### Other groups

| Group | Scope | Limit |
| --- | --- | --- |
| `inference_count` | api\_key | 300 / min |
| `model_list` | api\_key | 600 / min |

## Per-key versus global

Because per-key limit values are global group defaults, you cannot read a per-key rate from a key. All keys in a group are governed by the same default number. If you need a different rate for a particular workload, that is a deployment-level policy change, not a per-key setting.

## In-flight (concurrency) limits

`inference_generate` also bounds **concurrent in-flight requests**: 25 per org and 10 per API key. This is independent of the per-minute rate. Exceeding the concurrency bound is rate-limited the same way (a `429` with `Retry-After`) even if you are under the per-minute limit. If you fan out many long-running generations at once, the concurrency bound is the one you will hit first.

## What you get on a 429

- HTTP status `429`.
- A `Retry-After` header giving the seconds to wait.
- The [error envelope](/resources/errors-reference) with `code` either `rate_limited` or `insufficient_quota`.

**Only `Retry-After` is emitted.** There are **no `X-RateLimit-*` headers** (no `X-RateLimit-Limit`, `X-RateLimit-Remaining`, or `X-RateLimit-Reset`). Do not build client logic that reads a remaining-quota or reset header, because none is sent. Read the wait time from `Retry-After` and back off accordingly.

```http
HTTP/1.1 429 Too Many Requests
Retry-After: <seconds>
Content-Type: application/json; charset=utf-8

{"error":{"message":"...","code":"rate_limited"}}
```

## Fail-open versus fail-closed

If the limiter's own infrastructure errors (for example it cannot be consulted), most groups **fail open**: the request is allowed to proceed rather than be rejected on an infrastructure fault.

A specific **fail-closed** set rejects requests when the limiter cannot be consulted, because letting them through unbounded is riskier than a brief outage. The fail-closed groups are:

- `auth_interactive`
- `inference_generate`
- `inference_count`
- `mcp_*`
- `oauth`
- `org_admin`
- `spa_api_account_org`

For these groups, a limiter outage surfaces to the caller as a rejection rather than a silent pass-through. Everything outside this set fails open.

## Handling limits in your client

- Treat `429` as retryable, but only after the `Retry-After` interval. See [Retry behavior](/api-reference/errors-and-rate-limits/retry-behavior).
- Do not poll for a remaining-quota header; it does not exist.
- Bound your own concurrency to stay under the in-flight limits (25 per org, 10 per key on `inference_generate`).
- Distinguish `rate_limited` (transient, retry after wait) from `insufficient_quota` (a spend or usage cap, which may need a quota change) even though both return `429`.
