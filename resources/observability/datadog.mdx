The Nito gateway does not have a native Datadog integration. What it does have is an OpenTelemetry exporter that ships traces and metrics over OTLP. Datadog can ingest OTLP, so the supported pattern is **OTLP to Datadog** through a collector or the Datadog Agent, not a first-party Datadog client inside the gateway.

<Info>
  The gateway speaks OTLP. Datadog is one of several backends that can sit behind that OTLP stream. Nothing in the gateway is Datadog-specific.
</Info>

## The OTLP export

When observability is enabled, the gateway exports OpenTelemetry using these settings:

| Variable | Purpose |
| --- | --- |
| `OTEL_ENABLED` | Turns OpenTelemetry export on. |
| `OTEL_EXPORTER_OTLP_ENDPOINT` | The OTLP collector or agent endpoint. |
| `OTEL_EXPORTER_OTLP_PROTOCOL` | The OTLP protocol (for example gRPC or HTTP). |
| `OTEL_SERVICE_NAME` | The service name attached to emitted spans and metrics. |

These are deployment-side settings for whoever operates the gateway. Point `OTEL_EXPORTER_OTLP_ENDPOINT` at a collector that forwards to Datadog.

## The routing shape

```text
Nito gateway  --OTLP-->  OTLP-compatible collector / Datadog Agent  -->  Datadog
```

The gateway's job ends at the OTLP endpoint. The collector or agent is responsible for authenticating to Datadog and mapping OTLP resources onto Datadog's data model.

## Client-side correlation

Independent of the server-side OTLP stream, you can correlate calls in Datadog from the client by logging the `X-Request-ID` header and the `usage` block returned on every response. Ship those as Datadog logs or custom metrics from your own application so a client-side event lines up with a server-side trace.

An example endpoint would look like:

```text
POST https://YOUR-NITO-BASE-URL/api/v1/chat/completions
```
