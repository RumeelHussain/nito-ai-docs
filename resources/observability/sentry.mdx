The Nito gateway has real, built-in Sentry support for server-side error tracking. Unlike some of the other backends in this section, this is a first-party integration: the gateway itself reports errors to Sentry when configured to do so.

<Info>
  Sentry here is a deployment-side capability. It captures errors from inside the gateway process, not from your client. Enabling it is the job of whoever operates the Nito deployment.
</Info>

## Configuration

Two environment variables control it:

| Variable | Purpose |
| :-: | :-: |
| `SENTRY_ENABLED` | Turns Sentry error reporting on. |
| `SENTRY_DSN` | The Sentry DSN the gateway reports to. |

When `SENTRY_ENABLED` is set and a valid `SENTRY_DSN` is provided, the gateway sends server-side errors to that Sentry project. These live alongside the other observability settings (`OTEL_*`, `OPENOBSERVE_*`) in the gateway configuration.

## What is Captured

Sentry captures errors that occur inside the gateway. This is distinct from the structured error envelope your client receives on a failed request, which carries a `message` and a `code` (there is no `type` field). The client-facing error and the Sentry-side error report are two different views of the same failure.

To line up a client-observed failure with a Sentry event, log the `X-Request-ID` from the response. It is the shared identifier that lets an operator find the corresponding server-side error.

## What is Not Captured

Sentry is for errors and exceptions, not for request tracing or usage metering. For distributed tracing and metrics, use the OpenTelemetry export documented under [Datadog](/resources/observability/datadog). For per-call cost and volume, log the `usage` block from each response.
