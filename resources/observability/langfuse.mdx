There is no verified native Langfuse integration in the Nito gateway. The gateway does not send traces to Langfuse and has no Langfuse configuration. What you can do today is client-side LLM observability: instrument your own application to record each Nito call in Langfuse using the data every response returns.

<Warning>
  Nito does not push data to Langfuse. Any Langfuse tracing described here runs entirely in your application code, using values you already hold.
</Warning>

## What to Log Per Call

Every gateway response gives you the primitives Langfuse traces are built from:

- **`X-Request-ID`** - a stable identifier for the call, echoed from your header or generated by the gateway. Use it as the external ID on a Langfuse observation so you can trace a Langfuse record back to a specific gateway request.
- **The `usage` block** - `prompt_tokens`, `completion_tokens`, `total_tokens`, and optional detail such as `reasoning_tokens` and cached-token counts. Map these onto Langfuse's usage fields for cost and volume tracking.
- **The `model` field** and `finish_reason` - useful as Langfuse metadata.

## Prompts Stay on Your Side

Nito's default is metadata-only retention: prompts and completions are not persisted server-side. That means the only place a full prompt or completion exists for logging is your own client. If you want prompts and outputs in Langfuse, you capture them in your application before and after the call and send them yourself. This keeps you in control of exactly what leaves your environment.

## A Minimal Shape

```shellscript wrap
1. Start a Langfuse trace / generation in your app.
2. Send the request to https://nito.dev.zprotocol.org/api/v1/chat/completions
   (send your own X-Request-ID so the value is known up front).
3. On the response, record: X-Request-ID, model, usage.*, finish_reason,
   and any prompt/completion text you chose to keep client-side.
4. End the Langfuse generation.
```
