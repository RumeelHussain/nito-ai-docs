title: "S3"
description: "Exporting your own Nito call logs to S3-compatible object storage from the client. There is no first-party telemetry-to-S3 export in the gateway."
---

The Nito gateway uses S3-compatible object storage internally, configured with `BLOB_STORAGE_*`. That storage is for the gateway's own data, not a customer-facing telemetry export. There is no verified "export telemetry to S3" feature you can turn on as a caller. What you can do is export your own client-side logs to an S3 bucket you control.

<Warning>
  Nito does not write your telemetry to S3 for you. The `BLOB_STORAGE_*` configuration is internal to the gateway deployment. Any S3 export of Nito activity is something you build on your side.
</Warning>

## Client-Side Export

Because Nito's default is metadata-only retention, the authoritative record of your traffic is whatever you capture in your own application. To keep a durable log in S3:

1. On each response, capture the fields you care about: `X-Request-ID`, `model`, the `usage` block (`prompt_tokens`, `completion_tokens`, `total_tokens`, and detail fields), `finish_reason`, and timestamps.
2. Optionally include prompt and completion text, which only exists client-side under the metadata-only default.
3. Batch these records and write them to your own S3-compatible bucket (for example as newline-delimited JSON), using your own credentials and lifecycle rules.

This gives you a queryable archive without depending on any gateway-side export.

## Example

```text
POST https://nito.dev.zprotocol.org/api/v1/chat/completions
-> capture X-Request-ID + usage from the response
-> append to a local buffer
-> flush the buffer to s3://your-bucket/nito-logs/ on an interval
```
