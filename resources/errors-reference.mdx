This is the consolidated error catalog for Nito: the response envelope, every error kind the gateway emits, and the HTTP status each kind maps to. The [API reference error pages](/api-reference/errors-and-rate-limits/error-envelope) cover the same facts split by topic; this page is the single catalog you can scan end to end.

## The Error Envelope

Every gateway error response has this shape:

```json
{
  "error": {
    "message": "human-readable description",
    "code": "invalid_request",
    "param": "model"
  }
}
```

- `message` a human-readable description.
- `code` the error family (see the catalog below). Branch your handling on this.
- `param` present only when a specific field caused the error (for example `model`).

There is **no `type` field.** The error family is carried entirely in `code`. If you are porting from an API that keys on `error.type`, key on `error.code` here instead.

A few non-inference surfaces use different envelopes: OAuth endpoints follow RFC 6749 (`{"error","error_description"}`), and some browser/SPA endpoints use `{"error","code"}`. The `{"error":{"message","code","param?}}` shape above is the inference and general API envelope.

## Types of Errors

These are the only families the gateway emits, with `internal_error` as the fallback for anything unclassified.

| Code | Meaning |
| --- | --- |
| `invalid_request` | The request was malformed, failed validation, or used an unsupported feature or attachment. |
| `model_unavailable` | The requested model could not be resolved or is not available. `param` is `model`. |
| `unauthorized` | The credential is missing, malformed, revoked, or expired. |
| `forbidden` | Authenticated, but not permitted to perform this action. |
| `insufficient_quota` | The org or key is over a spend or usage quota. |
| `payment_required` | Payment is required before the request can proceed. |
| `provider_error` | An upstream provider returned an error. |
| `rate_limited` | A rate limit was exceeded. |
| `unavailable` | The service or an upstream is temporarily unavailable. |
| `internal_error` | Fallback for an unexpected or unclassified error. |

## HTTP Status Mapping

Each error kind maps to a fixed HTTP status. Note that two distinct kinds (`invalid_request` and `model_unavailable`) both map to `400`, and two (`insufficient_quota` and `rate_limited`) both map to `429`, so you must read `code` to tell them apart.

| Code | HTTP status |
| :-: | :-: |
| `invalid_request` | 400 |
| `model_unavailable` | 400 (`param` is `model`) |
| `unauthorized` | 401 |
| `forbidden` | 403 |
| `payment_required` | 402 |
| `insufficient_quota` | 429 |
| `rate_limited` | 429 |
| `unavailable` | 503 |
| `provider_error` | 502 |
| unknown / unclassified | 500 |

The unclassified fallback maps to `500` and surfaces as `internal_error` in the envelope.

## How to Respond

| Code | Recommended action |
| :-: | :-: |
| `invalid_request` | Fix the request. Do not retry unchanged. Read `message` and `param`. |
| `model_unavailable` | Pick another model. See [List models](/api-reference/models-catalog/list-models). |
| `unauthorized` | Check the `z_sk_` key and `Authorization` header. Do not retry unchanged. |
| `forbidden` | Check org role and permissions. Do not retry unchanged. |
| `insufficient_quota` | Wait or raise the quota, then retry after `Retry-After`. |
| `payment_required` | Add funds or payment, then retry. |
| `provider_error` | Transient. Retry with backoff. |
| `rate_limited` | Retry after `Retry-After`. See [Rate limits](/api-reference/errors-and-rate-limits/rate-limits). |
| `unavailable` | Transient. Retry with backoff. |
| `internal_error` | Retry with backoff; if it persists, contact support with the `X-Request-ID`. |

## Errors Mid-Stream

In a streaming (SSE) response, an error that occurs after the stream has opened arrives as a chunk with a top-level `error` object (`{message, code, param}`) and a choice whose `finish_reason` is `"error"`, rather than as an HTTP status. If the stream fails to open before any bytes are written, a normal HTTP error response is sent instead. See [Error events](/api-reference/streaming-format/error-events).

## Debugging with Request IDs

Every response carries an `X-Request-ID` (echoed from your request if you sent one, otherwise generated). Capture it and include it when reporting a problem, especially for `internal_error` and `provider_error`, so the issue can be traced. See [Request IDs](/api-reference/overview/request-ids).
