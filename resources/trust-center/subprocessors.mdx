A subprocessor list names the third parties that may process customer data on Nito's behalf, so that customers can review who is in the data path and be notified when it changes. A published list typically includes each party's name, the purpose of processing, and the location or region where processing occurs.

<Warning>
There is no authoritative subprocessor list in the codebase these docs are built from. The list below is not published and must not be treated as complete, official, or contractually binding. The authoritative list is owned by Legal.
</Warning>

## Technical integrations are not a subprocessor list

Several providers are named across these docs because they are technical integrations in the inference path: **xAI**, **OpenRouter**, **NEAR**, **Phala**, **fal**, and **Stripe** (billing). Naming an integration is an engineering fact, not a legal designation. Whether each of these is a subprocessor, in what role, and for what data category is a determination Legal makes, not something inferred from the fact that code calls the provider.

Do not present the list above as an official subprocessor register. It is context for how the system works, and it may be incomplete or framed differently once Legal defines the authoritative version.

## What the published list is expected to contain

Once authored by Legal, the subprocessor list is expected to include, per entry:

- The subprocessor's legal name.
- The service provided and the purpose of processing.
- The categories of data processed.
- The processing location or region.
- A change-notification mechanism for customers.

The list is also expected to align with the [DPA](/resources/trust-center/dpa), which governs how subprocessors may be engaged and how changes are communicated.
