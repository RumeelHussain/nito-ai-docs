This page explains how Nito prices inference and where to read the numbers. The per-model rates are configured in the billing system and surfaced on each model's catalog entry, so the authoritative figures come from the live catalog, not a static table here.

## How pricing works

Nito bills in credits, tracked internally in nanos (billionths of a unit) for precision. Inference is priced primarily per token, with separate rates for input and output, plus additional rates for cached input tokens, reasoning output tokens, and server-side tool usage. Image generation is priced by output. Before each call, Nito places a hold sized to the expected cost, then settles it against actual usage. Failed requests are not billed. See [Check usage and credits](/get-started/check-usage-and-credits).

## Where to read per-model cost

Pricing and context window are not on the model card by default. When pricing is configured for a model, its `GET /v1/models` entry carries a separate `pricing` object, joined from the billing system at response time. Its fields are expressed in nanos and cover input and output token rates, cache read and write rates, the reasoning output rate, web search and fetch rates, an image output rate, and the model's context limit. Read pricing from the entry so your estimates track the live configuration.

```bash
curl "https://nito.dev.zprotocol.org/v1/models" \
  -H "Authorization: Bearer $NITO_API_KEY"
# each entry includes a "pricing" object in nanos when configured
```

## Server-side tool pricing

Server-side tools such as web search and web fetch are billed in addition to the model tokens. Their per-use rates are part of the same pricing configuration. See [Tool pricing](/api-reference/server-tools/tool-pricing) for the reference.

## Estimating a call

A rough estimate for a text call is `input_tokens * input_rate + output_tokens * output_rate`, plus any reasoning or tool costs, all in nanos. Because rates are per model and configured live, pull them from the `pricing` object rather than hard-coding numbers. The actual charge for a completed call is always reported in the response `usage` block and aggregated in the org usage overview.

## Common mistakes

- **Hard-coding prices.** Rates are configured live and can change. Read them from the model entry.
- **Expecting `pricing` on every entry.** It appears only when configured for that model; absence means it is not yet set, not that the model is free.
- **Treating nanos as whole credits.** Nanos are billionths; divide by 1e9.
- **Forgetting tool and reasoning costs.** Web tools and reasoning tokens are billed on top of base input and output.
