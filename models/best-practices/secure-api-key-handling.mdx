---
title: "Secure API Key Handling"
description: "A z_sk_ key is full org-scoped access with no per-key scopes. Store it like a password, isolate workloads, cap spend, and rotate by revoking."
---

A Nito API key is more powerful than it looks. A `z_sk_` key grants full, organization-scoped access to your account. There are no per-key scopes or permissions and no "read-only" or "management-only" key type. Any valid key can reach the entire catalog and spend against your balance. That single fact drives every recommendation on this page: treat these keys exactly as you would treat a password with full account access, because that is what they are.

## Understand What the Key Is

When you authenticate, you send the key as a bearer token in the `Authorization` header. Nito resolves it to your organization, the account that owns billing and keys, and runs the request under that org's privileges. The key does not carry a narrower scope; privilege is controlled entirely by organization roles, not by anything encoded in the key itself. So the only meaningful boundaries you can draw are which workloads hold which keys, and what spend caps those keys carry.

One reassuring detail: keys are never stored in the clear on the server. Nito keeps only a one-way hash of each token, so the raw key exists in exactly one place after creation, wherever you saved it. You see the raw token once, at creation time, and never again. If you lose it, you rotate rather than recover.

## Store Keys in Secrets, Never in Code

The most common way keys leak is committing them to source control or shipping them to a browser. Do neither.

- **Use environment variables or a secrets manager.** Load the key from the environment at runtime (`NITO_API_KEY`) or from a dedicated secrets store. Keep it out of the repository entirely.
- **Never put a key in client-side code.** Anything that runs in a browser or a mobile app is readable by the user. A key embedded there is a public key. Route model calls through your own backend, which holds the key server-side.
- **Keep keys out of logs and error reports.** Never log the `Authorization` header. See [Privacy-safe logging](/models/best-practices/privacy-safe-logging).

## Isolate Workloads and Bound Blast Radius

Because a key cannot be scoped, isolation and spend caps are your main tools for limiting damage if one leaks.

- **One key per workload.** Give each service, job, or environment its own key. When something goes wrong, you can revoke that one key without taking down everything else, and you can tell from usage which workload a key belongs to.
- **Set per-key spend caps.** Each key supports a spend cap of type `daily`, `monthly`, or `overall`, and breaching the cap force-expires the key. A tight cap turns a leaked key from an open-ended liability into a bounded one. Size the cap to what the workload should legitimately spend. See [Cost control](/models/best-practices/cost-control).

## Rotate by Revoking and Re-Creating

There is no key update or edit operation. The key lifecycle is create, list, revoke, and revocation is a soft delete. So rotation is simply: create a new key, deploy it, then revoke the old one.

Key management — creating, listing, and revoking keys — is done from your Nito dashboard, authenticated by your browser session rather than a `z_sk_` API key. So rotation is a dashboard action, not an API call: mint a new key, deploy it into your secrets store, then revoke the old one.

Rotate on a schedule, and rotate immediately if you suspect exposure. Because you capture the raw token only once at creation, read it straight into your secrets store rather than pasting it around.

## Control Who Can Manage Keys

Key management is restricted to organization roles. Only `owner` and `admin` can create, list, or revoke keys; a `member` cannot. Use those roles deliberately: keep the set of people who can mint keys small, and review membership when people join or leave. This is the human side of blast-radius control, and it is the only access-control lever the key model gives you.

## A Quick Checklist

- Keys in env vars or a secrets manager, never in source or client-side code.
- One key per workload, each with a spend cap.
- Rotate by creating new, then revoking old; rotate on suspicion immediately.
- `Authorization` header never logged.
- Key management limited to owner and admin roles.
