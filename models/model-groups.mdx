A single model can often be served in more than one place: the same model offered by two providers, or the same model available at different privacy levels. Nito hides that multiplicity behind one name. A model group is the gateway's way of collapsing several equivalent backends into one logical choice, so that you name a model once and Nito routes your call to whichever backend is the best fit at that moment. This is what makes selection resilient: you commit to a name, not to a particular machine.

If that sounds abstract, here is the everyday version. Think of a model group like a phone number that rings several desks. You dial one number; whoever is free and qualified picks up. You never have to know who answered, only that your call connected.

## What a group is

A group gathers several physical models that share one canonical identity. A physical model is a specific model on a specific provider, identified internally as `provider/providerModelID`. A canonical identity is the single public name they all answer to. Grouping them gives you three properties:

- **You select by the logical model ID, not by a specific backend.** The logical ID is the public name you put in your request.
- **One logical ID can fan out into per-privacy-mode options.** A group can expose ordinary, private, `:confidential`, and `:encrypted` variants, depending on which backends in the group actually support which modes. A backend only shows up under a mode it can genuinely serve.
- **If one backend is unavailable, another in the group can serve the call.** Health is not your problem to track.

The grouping is computed per surface, so the chat catalog and the image catalog each group independently.

## Listing groups

```bash
curl "https://nito.dev.zprotocol.org/api/v1/model-groups?capability=chat" \
  -H "Authorization: Bearer $NITO_API_KEY"
```

The `capability` query parameter selects the surface: `chat` (the default) or `images`. The response is a set of groups, each with a label and a list of options, where each option corresponds to a logical ID for a supported privacy mode. An optional org context can scope the result.

This endpoint is a convenience for **building a model picker** — a labeled, grouped list for a dropdown or settings screen. It is not how routing happens: you never call it to route a request. Routing is automatic and per-request — you name a logical ID in your `model` field and Nito selects the backend (see [Backend selection](/models/backend-selection)).

## How aliasing actually works

People often expect an "alias" to be a manual mapping someone maintains, like a spreadsheet that says "this nickname means that model." Grouping is what aliasing is in practice, and it is automatic. Many provider-scoped rows that share a canonical model ID are presented as one option, keyed by the logical ID for each supported privacy mode. You do not register or manage aliases yourself. You benefit from them by naming the logical model and letting the group resolve to a concrete backend.

A worked example makes the collapse concrete. Suppose a popular chat model is available from two providers, and one of those providers also offers it inside a TEE. That is three physical rows. The group collapses them into a small set of logical options: an ordinary-mode option that either provider can serve, and a `:confidential` option that only the attestation-capable provider can serve. You see clean choices; Nito keeps the messy provider list behind them.

## Fallbacks, and the one place they stop

When you call a logical model, Nito gathers the candidate backends in the group and chooses one. If the preferred backend is unavailable, the next eligible candidate is used. There is also a second kind of fallback, the provider-default fallback: if the catalog has no exact match for a bare logical ID, Nito can route to a provider's configured default model.

The provider-default fallback applies to the ordinary, anon, and private modes only. It does not apply to `:confidential` and `:encrypted`. The reason follows directly from what attestation means. An attestation is a signed statement that your call ran inside a specific sealed environment. If Nito quietly substituted a default backend for a `:confidential` request, any attestation you later fetched would describe a different environment than the one you asked for, which defeats the purpose. So for `:confidential` and `:encrypted` there is no silent substitute: either an attestation-capable backend in the group can serve the call, or the call fails.

The exact ordering Nito uses to choose among the eligible candidates is described in [Backend selection](/models/backend-selection).

## Why this matters

Model groups are what let you write `model: "<logical-id>"` once and stop worrying about which provider is healthy, which backend is cheapest, or which one satisfies your privacy requirement. The gateway turns one name into a resilient routing decision, made fresh on every call. The cost of that convenience is that a logical ID is not a fixed machine, which is exactly the point of the next two common mistakes.

## Common mistakes

- **Expecting a `:confidential` call to fall back to an ordinary backend.** It will not. A `:confidential` call that can't be served in a sealed (attested) environment **fails** rather than quietly downgrading — the provider-default fallback is deliberately excluded from `:confidential` and `:encrypted`, so an attestation always describes the environment you actually asked for.
- **Treating a logical ID as a single fixed backend.** It can resolve to different backends across calls based on availability and cost. For attested calls, the attestation tells you which backend actually ran.
- **Listing the wrong surface.** The `model-groups` endpoint is per capability; pass `?capability=images` to see image groups rather than the `chat` default.
