---
title: What Nito Does Not Defend Against
description: A frank statement of Nito's privacy boundaries, so you design around the system's real limits rather than an imagined version of it.
---

This page is the half that most documentation leaves out. It states, specifically, the things Nito does **not** protect you from, so you can decide where you need to add your own controls. None of this is a confession of a bug. Each item is a deliberate scope boundary, and naming it is what makes the defenses on the [companion page](/privacy/threat-model/what-nito-defends-against) trustworthy. A reviewer who knows exactly where a system stops can rely on what it claims; a reviewer kept in the dark cannot.

If you read one idea from this page, read this one: **the gateway gives you fast, baseline-verified answers and hands you the evidence to finish the verification; the final, deepest checks are yours to run.**

## The gateway does not perform full hardware-quote verification

When you use `tee` or `e2e-tee`, the gateway verifies the attestation at a **baseline** level: it confirms the evidence is present and well-formed, that the proof format and signing material are there, that your nonce is echoed, and that the `report_data == signing_address + nonce` binding holds. What it deliberately does **not** do is verify the underlying hardware quote. The gateway does not validate Intel TDX quotes, NVIDIA GPU evidence, certificate chains, certificate revocation lists (CRLs), or TCB (trusted computing base) policies.

That full hardware-quote verification is the **client's responsibility**. The gateway returns the raw evidence, signing address, model public key, proof format, and echoed nonce precisely so you can run those checks yourself, and there is a reference verifier to model the process.

**What this means for you:** if your threat model requires hardware-rooted assurance, you must run the client-side verification. A successful `:tee` response means the call was baseline-verified and the raw evidence is available, not that the gateway has fully vouched for the silicon. This is why these docs say **attested** and **verifiable**, never "trustless" or "fully verified."

## Anti-replay depends on the caller supplying a nonce

The freshness of an attestation is bound through the nonce you send: `report_data = signing_address + nonce`. That binding is only as strong as the nonce. Anti-replay is **caller-driven**. If you do not supply a fresh, random nonce, the binding can be satisfied by an empty value, which is not replay protection.

**What this means for you:** to get replay protection, you must generate a fresh random nonce per attestation request and confirm it is echoed back. The gateway does not invent one for you, and an omitted nonce is not an error, it is simply a weaker, unbound report. Treat the nonce as a required input whenever freshness matters.

## There is no chat-prompt moderation or prompt-injection classifier

The gateway does not scan your chat prompts for unsafe content, and it does not run a prompt-injection or jailbreak classifier on the conversation. The only content-filtering that exists is the **outbound web-search query redaction**, which strips secrets and PII from a search query before it leaves to the search backend. That is a narrow, outbound defense on one path. It is not a guardrail over the chat itself.

**What this means for you:** if your application needs moderation, injection defense, or output filtering, you build it. The gateway routes the conversation; it does not police its content. Do not assume an injected instruction inside a document or a tool result will be caught, because there is no classifier looking for it.

## A non-TEE call is not sealed from the provider

For ordinary and private modes, your call runs on the provider's normal infrastructure. Nito constrains it with a per-request no-retention setting (`Store: false`, a ZDR policy, or `X-Fal-Store-IO: 0`), but that is an instruction to the provider, not a sealed environment. The call is **not** isolated from the provider beyond the provider honoring that retention setting. There is no attestation, no hardware isolation, and no proof for a non-TEE call.

**What this means for you:** if you need the provider to be unable to see the data, rather than instructed not to keep it, you must use `tee` or `e2e-tee` on a supporting model. Private mode is a strong retention posture, not a sealing guarantee. The distinction is the whole reason the privacy dial exists.

## Summary

| Boundary | Where it stops | Your responsibility |
|---|---|---|
| Attestation verification | Baseline only; no TDX/GPU/cert-chain/CRL/TCB checks | Run the client-side hardware verification |
| Anti-replay | Only as strong as the nonce you send | Supply a fresh random nonce; confirm the echo |
| Content safety | Only outbound web-search query redaction | Add moderation / injection defense yourself |
| Non-TEE isolation | Retention setting only, no sealing | Use `:tee` / `:e2e-tee` for hardware isolation |
