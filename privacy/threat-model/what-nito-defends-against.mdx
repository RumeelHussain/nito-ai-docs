This page lists the defenses Nito actually implements, and only those. Every item is tied to a specific mechanism in the system, not to a posture or an aspiration. Read it alongside its companion, [What Nito does not defend against](/privacy/threat-model/what-nito-does-not-defend-against); a defense is only meaningful next to its boundary.

If you read one idea from this page, read this one: **each defense below is a concrete behavior of the gateway, chosen so you can verify it rather than take it on faith.**

## Default retention is metadata only

The gateway's default posture, even for an ordinary call, is to keep metadata only. Prompts, completions, reasoning, tool calls, and uploaded file contents are request-local: they are used to serve the request and are not persisted by the gateway. Telemetry is restricted to a safe allowlist of fields, and inference responses are served with `Cache-Control: no-store`. Where a chat history exists, for example in a browser app, it lives encrypted on the client side, not on the gateway.

**What this defends against:** a gateway-side store of your prompts and replies accumulating over time. There is no such store to breach, subpoena, or mine, because content is not written down in the first place.

## Provider-side no-retention enforcement

The metadata-only default would mean little if the provider kept everything once the call left the gateway. So Nito constrains each provider hop with an explicit per-request setting: `Store: false` for xAI, a zero-data-retention policy for OpenRouter (with non-ZDR models never imported), and `X-Fal-Store-IO: 0` for fal image requests. The instruction to not retain is sent on each call, not assumed from a default.

**What this defends against:** the boundary problem, where privacy stops at the edge of the gateway. The constraint travels with the request to the provider. The full per-provider detail is in [Provider-side enforcement](/privacy/provider-side-enforcement).

## Sealed, attested execution for the top tiers

For `confidential` and `encrypted` modes on NEAR and Phala, the call runs inside a Trusted Execution Environment, a hardware-isolated, sealed compute box the host cannot peer into, and the provider returns an **attestation**: a signed statement about the environment that ran the call. You can fetch that attestation and verify it. The gateway performs baseline verification (evidence present and well-formed, proof format and signing material present, your nonce echoed, and the `report_data == signing_address + nonce` binding holds), and hands you the raw evidence to verify the hardware quote yourself.

**What this defends against:** having to take the provider's word that your call ran where it claimed. The result is **attested** and **verifiable**. (The boundary, that the gateway's verification is baseline and not full hardware-quote verification, is on the [companion page](/privacy/threat-model/what-nito-does-not-defend-against) and is essential to read.)

## Credentials are stored only as hashes

API keys and tokens are never stored in raw form. A token is 32 random bytes, and what the gateway persists is an HMAC-SHA256 of the token under a server secret. The same is true across stored credentials: they are kept as hashes, not as recoverable secrets.

**What this defends against:** credential theft from the datastore. A leak of the stored values does not hand an attacker usable keys, because the raw tokens were never written down.

## Outbound web-search queries are redacted

When a request triggers web search, the query that goes to the search backend is passed through a redaction step that strips secrets and personally identifiable information first. The query the model formed is cleaned before it leaves to the search provider.

**What this defends against:** leaking secrets or PII into a third-party search backend through a tool call. It is a targeted, outbound defense on the web-search path. Note its scope precisely: it redacts _outbound web-search queries_. It is not a general moderation or prompt-injection defense, which the [companion page](/privacy/threat-model/what-nito-does-not-defend-against) covers.

## Standard web security controls

The gateway ships the ordinary, expected web-security hardening rather than leaving it to chance:

- A full set of security response headers: `X-Content-Type-Options: nosniff`, `X-Frame-Options: DENY`, a `Referrer-Policy`, cross-origin isolation headers (COOP/CORP), a `Permissions-Policy`, and a `Content-Security-Policy`. HSTS is set when the base URL is HTTPS.
- CSRF protection on browser-session endpoints via a double-submit cookie (`ig_csrf`) compared against an `X-CSRF-Token` header using a constant-time comparison.
- URL attachments must be HTTPS, carry no credentials, and may not point at private, loopback, or metadata hosts.

**What this defends against:** the common web-layer attacks, clickjacking, MIME sniffing, cross-site request forgery, and server-side request forgery through attachment URLs.

## Summary

| Defense | Mechanism | Guards against |
| --- | --- | --- |
| Metadata-only retention | Content request-local; `Cache-Control: no-store` | A gateway-side prompt/reply store |
| Provider no-retention | `Store:false` / ZDR / `X-Fal-Store-IO:0` per request | Retention past the gateway boundary |
| Sealed, attested TEE | NEAR/Phala TEE \+ verifiable attestation | Taking provider placement on faith |
| Hashed credentials | HMAC-SHA256 of 32-byte tokens | Credential theft from the datastore |
| Outbound query redaction | Secret/PII stripping before search | Leaking PII into the search backend |
| Web security controls | Security headers, CSRF, attachment-URL rules | Clickjacking, CSRF, SSRF, MIME sniffing |
