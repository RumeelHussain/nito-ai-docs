description: "The top tier. Client-side encryption to the sealed environment on top of TEE, with the most constraints."

T4 Encrypted is the top rung of the [privacy ladder](/privacy/tiers) and the strongest privacy Nito offers. In the code it is the `e2e-tee` mode, selected with the `:e2e-tee` suffix plus a set of encryption headers. It keeps everything [T3](/privacy/tiers/t3-private-plus-tee) gives you, the sealed and attestable execution of a Trusted Execution Environment, and adds end-to-end encryption: your data is encrypted client-side to the sealed environment itself. In exchange, T4 carries the most constraints of any tier. It is the choice for maximum assurance, used only where the call genuinely needs it.

## What it adds over T3

T3 runs your call inside a TEE and lets you verify it ran there. T4 keeps that and adds **client-side encryption to the sealed environment**. Rather than relying on transport security up to the gateway, your data is encrypted so that it can be decrypted only inside the sealed TEE that runs the computation. That is what "end-to-end encrypted TEE" means: the protection extends from your client all the way to the attested environment, not just to the gateway's edge.

This is the maximum-assurance posture. It is also the most constrained, because encrypting end-to-end to a sealed box rules out the gateway features that would need to read or transform your request in flight.

## What it requires

T4 is available on TEE providers only, **NEAR and Phala**, and only for models whose catalog entry reports `e2ee_supported: true`. The gateway's invariants require end-to-end encryption to sit on a `tee`-tier model that has attestation and an encryption subject, so a model must already qualify for T3 before it can offer T4. Confirm support on the catalog entry:

```bash
curl https://YOUR-NITO-BASE-URL/v1/models \
  -H "Authorization: Bearer $NITO_API_KEY"
```

Look for `e2ee_supported: true`. If it is absent, the model cannot serve T4 and the call will be rejected rather than served without encryption.

## The constraints (read before you build)

T4 is non-streaming and supports a deliberately narrow feature set. When the encryption headers are present, the gateway rejects:

- **Streaming.** End-to-end encrypted TEE is **non-streaming** by design. A request that asks to stream is rejected rather than quietly served without encryption.
- **Reasoning.**
- **Tools.**
- **Prompt caching.**
- **Multi-part messages.**

It is also restricted to the **NEAR and Phala** providers. If you need any of the features above, or you need streaming, use [T3](/privacy/tiers/t3-private-plus-tee) instead, which keeps attested execution while supporting streaming and the normal feature set. T4 trades those away for the strongest protection.

## How to select it

Append `:e2e-tee` to the model ID **and** supply the `X-Z-TEE-*` encryption headers. Both are required: the suffix alone is not enough, and the headers without a supporting model are rejected.

```bash
curl https://YOUR-NITO-BASE-URL/v1/chat/completions \
  -H "Authorization: Bearer $NITO_API_KEY" \
  -H "Content-Type: application/json" \
  -H "X-Z-TEE-...: ..." \
  -d '{
    "model": "MODEL-ID:e2e-tee",
    "stream": false,
    "messages": [{"role": "user", "content": "Process this highly sensitive content."}]
  }'
```

`:e2e-tee` is one of only two valid suffixes (the other is `:tee`). If the model does not support end-to-end encrypted TEE, the call fails; a privacy mode never silently downgrades to a weaker tier.

## Verifying the environment

Because T4 sits on top of TEE, the same attestation story applies: the sealed environment can produce a signed statement you can verify, the gateway performs a baseline verification, and the raw evidence is returned for full client-side verification. The attestation endpoint accepts the `e2e-tee` trust tier as well. The mechanics, the response fields, and a reference verifier are in [TEE attestation](/privacy/tee-attestation), and the verification trade-offs described for [T3](/privacy/tiers/t3-private-plus-tee) apply unchanged here.

## When to choose it

Choose T4 only when the call needs the maximum assurance that nobody between your client and the sealed environment can read the data, and when you can live without streaming, reasoning, tools, prompt caching, and multi-part messages. That is a narrow set of cases: the most sensitive single-shot requests where end-to-end confidentiality outweighs every convenience. For everything else, including most interactive and feature-rich work, [T3](/privacy/tiers/t3-private-plus-tee) is the stronger practical choice.
