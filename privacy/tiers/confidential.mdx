Confidential is the third rung of the [privacy ladder](/privacy/tiers) and the first one backed by hardware you can check. It is the `confidential` tier, selected with the `:confidential` suffix on the model ID. It keeps the guarantees of the tiers below it and adds the one thing none of them can: your call runs inside a sealed environment, and you can obtain a signed statement that proves it did.

## Runs inside a TEE

Confidential runs your call inside a **Trusted Execution Environment (TEE)** — a hardware-sealed environment the host cannot read into, which produces a signed **attestation** you can verify. New to the concept? Start with the [TEE primer](/privacy/tee-attestation/what-a-tee-is).

That attestation is what separates Confidential from [Private](/privacy/tiers/private). Private holds the provider to zero-data-retention, but you take that on the provider's word. Confidential gives you an artifact you can independently check. What Confidential provides is attested and verifiable, and the gateway's verification is a baseline rather than full hardware verification (see below).

## What does the Confidential Tier Requires

Confidential is offered by specific providers only, so not every model can run in a TEE. A model supports it when its catalog entry reports `attestation_available: true` (with `privacy_tier: "confidential"`). The TEE providers are **NEAR** and **Phala**; the gateway's invariants require a `confidential`-tier model to name one of them as its TEE provider, and require attestation to be present for the tier. Filter the catalog for attestation support before you rely on it:

```bash
curl https://nito.dev.zprotocol.org/v1/models \
  -H "Authorization: Bearer $NITO_API_KEY"
```

Look for `attestation_available: true` on the entry. If it is absent, the model has no TEE backend and appending `:confidential` will be rejected rather than served at a weaker tier.

## How to Select the Confidential Tier

Append `:confidential` to the model ID. Nothing else in the request body changes.

```bash wrap
curl https://nito.dev.zprotocol.org/v1/chat/completions \
  -H "Authorization: Bearer $NITO_API_KEY" \
  -H "Content-Type: application/json" \
  -d '{
    "model": "openai/gpt-oss-120b:confidential",
    "messages": [{"role": "user", "content": "Analyze this sensitive document."}]
  }'
```

`:confidential` is one of only two attested suffixes (the other is `:encrypted`). If the model does not support it, the call fails rather than quietly downgrading to a weaker tier.

## Streaming Works at Confidential Tier

A key reason to choose Confidential over [Encrypted](/privacy/tiers/encrypted): **streaming is supported in `:confidential`**. You can attest a call and still receive tokens incrementally over a long-lived connection, which a chat UI needs to show output as it arrives. Encrypted's end-to-end encryption, by contrast, is non-streaming by design. So when you need a verifiable execution boundary together with streaming, Confidential is the rung to use.

## Fetching and Verifying the Attestation

The attestation is what makes Confidential verifiable. You fetch it from a dedicated endpoint:

```bash
curl "https://nito.dev.zprotocol.org/api/v1/attestation?model=openai/gpt-oss-120b:confidential&nonce=YOUR-NONCE" \
  -H "Authorization: Bearer $NITO_API_KEY"
```

> **TODO (needs SME):** `https://nito.dev.zprotocol.org` is the devnet URL; confirm the canonical production base URL before public launch. (`YOUR-NONCE` is a fresh random nonce you generate per request.)

The gateway resolves the model to a physical attestation-capable backend, asks the provider for its attestation, and runs a **baseline verification** before returning. The baseline checks that the provider evidence is present and parseable, that the proof format and signing material are present, that your nonce matches the one observed from the provider, and that the report-data binding holds. It then returns the result, including the raw `evidence`, so you can perform **full client-side verification yourself**.

This split matters and is load-bearing: the gateway's baseline does **not** verify Intel TDX quotes, NVIDIA GPU evidence, certificate chains, CRLs, or TCB policies. Those are the client's responsibility, which is exactly why the raw evidence is handed back to you. Treat the gateway result as "baseline-verified" and do the hardware-level verification on your side for maximum assurance. The freshness guarantee is caller-driven: supply a fresh random nonce on each request so the attestation is bound to your call. The full procedure, the response fields, and a reference verifier are covered in [TEE attestation](/privacy/tee-attestation).

## Trade-offs of the Confidential Tier

- **Fewer backends.** The TEE comes from NEAR and Phala only, so Confidential reaches fewer models than Anonymous or Private, which can affect latency and availability.
- **A proof you must use.** The attestation is only worth as much as the verification you do with it. The gateway gives you a baseline and the raw evidence; full assurance comes from verifying the evidence client-side.
- **Streaming kept, encryption not yet added.** Confidential is the strongest tier that still streams. If you also need your data encrypted to the sealed environment, climb to [Encrypted](/privacy/tiers/encrypted) and accept its non-streaming constraint.

## When to Choose the Confidential Tier

Choose Confidential for sensitive calls where you need to verify that a sealed environment ran the work, and where you still want streaming and the normal feature set. It is the sweet spot for high-sensitivity interactive use: attested execution without giving up token-by-token output.
