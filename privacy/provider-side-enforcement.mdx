title: "Provider-Side Enforcement"
description: "How Nito constrains upstream providers so privacy holds past the gateway boundary, through concrete per-provider request settings."
---

A gateway can promise not to keep your data, but the call still leaves the gateway and runs on someone else's hardware. So the honest question a reviewer asks is not "what does Nito store?" but "what happens to my prompt once it crosses into the provider?" This page answers that. The short version is that Nito does not rely on a provider's general reputation or a marketing page. For each provider it sends a concrete, per-request setting that constrains what the upstream is allowed to do, and it refuses to import backends that cannot be constrained that way. Privacy holds past Nito's own boundary because of these specific switches, not in spite of the boundary.

If you read one idea from this page, read this one: **the gateway's metadata-only default is the floor, and provider-side enforcement is what keeps that floor intact after the request leaves the gateway.**

## Why the boundary matters

The gateway's own retention posture is metadata only. Prompts, completions, reasoning, tool calls, and uploaded file contents are request-local and are not persisted by the gateway. That covers everything inside Nito. It says nothing, on its own, about the provider that actually runs the model. Without provider-side enforcement, a private-tier promise would stop exactly where it matters most. So Nito treats the provider call as a hop that must carry its own enforcement, expressed as a setting on each outbound request.

The mechanism differs by provider because each provider exposes a different control. What is constant is that every provider Nito serves is constrained by an explicit, checkable setting.

## xAI (grok): storage disabled at the provider

For xAI-served models, Nito forces `Store: false` on every request through the OpenAI Responses adapter. This is the provider's own control for telling it not to retain the request and response. It is set by the gateway, not left to a default, so the provider is instructed on each call not to store the exchange.

The effect: an xAI call is told, request by request, not to keep your data at the source.

## OpenRouter: zero-data-retention by construction

OpenRouter is a router in front of many models, so Nito takes a stricter posture there and enforces it in two places at once: what gets imported, and how each request is flagged.

- **Only zero-data-retention endpoints are imported.** Non-ZDR models are never added to the catalog in the first place. If a model cannot be served under a zero-data-retention data policy, it is not reachable through Nito at all.
- **A ZDR data policy is applied per request.** The zero-data-retention flag is set so the upstream is told not to retain the exchange.
- **Embeddings and image requests always force ZDR.** These paths do not get the relaxation described next; they are pinned to zero retention.
- **The one relaxation is narrow and explicit.** The ZDR flag is relaxed only when OpenRouter's own server-side web tools are used on the request, because those tools require the provider to process the query to run the search. Outside that specific case, ZDR stays on.

So an OpenRouter route is zero-data-retention by construction: a non-conforming model never enters the catalog, and a conforming model is flagged ZDR on every request except the narrow, named web-tool case.

## fal (images): store flag off

Image generation is served by fal. Nito sends `X-Fal-Store-IO: 0` on image requests, which is fal's control for not storing the input and output of the call. As with xAI, this is set by the gateway on the request rather than relying on a provider default.

## NEAR and Phala: sealed execution with attestation

NEAR and Phala are the providers behind the `tee` and `e2e-tee` modes, and their enforcement is categorically stronger than a no-store flag. Instead of asking the provider not to keep your data, these run the call inside a Trusted Execution Environment, a hardware-isolated, sealed compute box, and produce an **attestation**: a signed statement about the environment that ran the call.

The attestation carries a provider-specific subject so you can confirm which sealed environment served you:

- **NEAR** identifies the environment by **domain**, with a default proof format of `near-attestation-report-v1`. The subject is optional.
- **Phala** identifies the environment by **appid**, with a default proof format of `redpill-attestation-report-v1`. The subject is required.

This is the difference between a promise and a proof. For xAI, OpenRouter, and fal, you are relying on a setting the gateway sends and the provider honors. For NEAR and Phala, you can fetch the attestation and verify, independently, that the call ran in the sealed environment it claims. The mechanics of that verification, and its deliberate limits, are covered in [Privacy architecture](/privacy/privacy-architecture) and the [threat model](/privacy/threat-model).

<Warning>
  Attestation is **baseline-verified** at the gateway and **verifiable** by you. The gateway does not perform full hardware-quote verification (Intel TDX quotes, NVIDIA GPU evidence, certificate chains, CRLs, or TCB policies). That deeper verification is the client's job. See [What Nito does not defend against](/privacy/threat-model/what-nito-does-not-defend-against).
</Warning>

## How the pieces fit

| Provider | What Nito sends or requires | What it constrains |
| --- | --- | --- |
| xAI (grok) | `Store: false` per request | Provider told not to retain the exchange |
| OpenRouter | ZDR-only import \+ per-request ZDR policy (embeddings/images always ZDR) | Non-conforming models never reachable; retention off except the named web-tool case |
| fal (images) | `X-Fal-Store-IO: 0` | Provider told not to store image input and output |
| NEAR | Sealed TEE \+ attestation, subject = `domain` | Verifiable sealed execution |
| Phala | Sealed TEE \+ attestation, subject = `appid` (required) | Verifiable sealed execution |

The throughline: a privacy claim that ends at the gateway boundary is not a privacy claim. Nito makes each provider hop carry an explicit setting, refuses to import backends that cannot be constrained, and for the top tiers replaces the setting with an attestation you can check yourself.
