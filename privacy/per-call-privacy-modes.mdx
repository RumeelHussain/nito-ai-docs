This page is the how-to for the idea introduced in the [Privacy overview](/privacy/overview): on Nito you choose privacy per request, and you choose it through the `model` string. There is no privacy field in the request body and no account-level privacy switch. The model ID carries two decisions at once, which model and how private, and this page shows exactly how to spell each mode and what each spelling requires.

## The suffix mechanism

The `model` value is parsed into a base model ID and an optional suffix:

- **No suffix** means ordinary or private mode. Which one you get depends on the model's own stored tier. A model whose tier is private resolves to private; otherwise it is ordinary. You do not type `:ordinary`, `:anon`, or `:private`; you simply send the plain model ID and the model's tier decides.
- **`:tee`** asks for execution inside a Trusted Execution Environment.
- **`:e2e-tee`** asks for an end-to-end encrypted TEE.

The **only** valid suffixes are `:tee` and `:e2e-tee`. Anything else, `:standard`, `:zdr`, `:e2ee`, `:anon`, `:private`, and so on, is rejected. The router will return an error to the effect that the logical model ID must use `<model>`, `<model>:tee`, or `<model>:e2e-tee` form. If you want a different mode, you change the model string and nothing else.

<Warning>
There is no separate `tier` parameter in the request body. Privacy is selected only by the model-ID suffix, and only `:tee` and `:e2e-tee` are valid. Sending any other suffix, or trying to pass privacy as a body field, is an error.
</Warning>

## Modes never silently downgrade

Before the examples, internalize the one rule that makes these modes trustworthy: **a privacy mode never silently downgrades.** If you ask for `:tee` and no TEE-capable backend can serve the call, the request fails. It does not quietly fall back to an ordinary backend. The same holds for `:e2e-tee`. A successful response to a suffixed request is therefore proof that the request was served at the mode you asked for, not a weaker one.

## Worked example: ordinary mode

Send the plain model ID, no suffix, on an ordinary-tier model. You get standard routing with the metadata-only default.

```bash
curl https://YOUR-NITO-BASE-URL/v1/chat/completions \
  -H "Authorization: Bearer $NITO_API_KEY" \
  -H "Content-Type: application/json" \
  -d '{
    "model": "grok-4.3",
    "messages": [{"role": "user", "content": "Summarize this release note."}]
  }'
```

## Worked example: private mode

Private mode is also selected with **no suffix**. The difference from ordinary is the model: send a model whose stored tier is private, and the route sets `RequireZeroDataRetention`, which drives provider-enforced ZDR. There is nothing extra to add to the request; the model's tier does the work.

```bash
curl https://YOUR-NITO-BASE-URL/v1/chat/completions \
  -H "Authorization: Bearer $NITO_API_KEY" \
  -H "Content-Type: application/json" \
  -d '{
    "model": "a-private-tier-model",
    "messages": [{"role": "user", "content": "Draft a reply to this customer email."}]
  }'
```

To find out which models are private-tier, consult the catalog (`GET /v1/models`) and read each entry's `privacy_tier`. Tiers are a property of the model, discovered at runtime, not something you hard-code.

## Worked example: TEE mode

Add `:tee` to a model that supports it. The call runs inside a sealed environment, and you can fetch an attestation afterward to verify it really did.

```bash
curl https://YOUR-NITO-BASE-URL/v1/chat/completions \
  -H "Authorization: Bearer $NITO_API_KEY" \
  -H "Content-Type: application/json" \
  -d '{
    "model": "grok-4.3:tee",
    "messages": [{"role": "user", "content": "Analyze this contract clause."}]
  }'
```

TEE-capable backends are a subset of the catalog (NEAR and Phala). If you request `:tee` on a model that has no TEE-capable backend, the call fails rather than downgrading. To verify the attestation for a TEE call, use `GET /api/v1/tee/attestation` with the model and a fresh random nonce; see [Privacy architecture](/privacy/privacy-architecture) for what the baseline verification checks and how to run the full client-side verification yourself.

## Worked example: end-to-end encrypted TEE mode

`:e2e-tee` is the strongest mode and has the most constraints. It adds client-side encryption so your content is encrypted to the sealed environment itself. Selecting it requires **both** the suffix **and** a set of `X-Z-TEE-*` encryption headers on the request:

| Header | Role |
|---|---|
| `X-Z-TEE-E2EE` | Marks the request as end-to-end encrypted. |
| `X-Z-TEE-Client-Pub-Key` | The client's public key. |
| `X-Z-TEE-Model-Pub-Key` | The model environment's public key. |
| `X-Z-TEE-Signing-Algo` | The signing algorithm in use. |
| `X-Z-TEE-Encryption-Version` | The encryption scheme version. |
| `X-Z-TEE-E2EE-Nonce` | A nonce for the encrypted exchange. |
| `X-Z-TEE-E2EE-Timestamp` | A timestamp for the encrypted exchange. |

Two hard constraints come with this mode:

- **It is non-streaming.** A request that sets the E2EE headers and also asks to stream is rejected (`streaming is not supported with E2E-TEE`). Reasoning, tools, prompt caching, and multi-part messages are likewise rejected in this mode.
- **It is supported on NEAR and Phala only.** No other provider serves `:e2e-tee`.

```bash
curl https://YOUR-NITO-BASE-URL/v1/chat/completions \
  -H "Authorization: Bearer $NITO_API_KEY" \
  -H "Content-Type: application/json" \
  -H "X-Z-TEE-E2EE: true" \
  -H "X-Z-TEE-Client-Pub-Key: <client-public-key>" \
  -H "X-Z-TEE-Model-Pub-Key: <model-public-key>" \
  -H "X-Z-TEE-Signing-Algo: <signing-algo>" \
  -H "X-Z-TEE-Encryption-Version: <version>" \
  -H "X-Z-TEE-E2EE-Nonce: <nonce>" \
  -H "X-Z-TEE-E2EE-Timestamp: <timestamp>" \
  -d '{
    "model": "a-near-or-phala-model:e2e-tee",
    "stream": false,
    "messages": [{"role": "user", "content": "Process this sensitive record."}]
  }'
```

## Choosing among the modes

| You send | Mode | Notable constraints |
|---|---|---|
| `model-id` (ordinary-tier model) | ordinary | Metadata-only default. |
| `model-id` (private-tier model) | private | Provider-enforced ZDR. |
| `model-id:tee` | tee | TEE-capable backends only; attestation available. |
| `model-id:e2e-tee` | e2e-tee | NEAR/Phala only; non-streaming; no tools/reasoning/caching/multi-part; `X-Z-TEE-*` headers required. |

Match the mode to the sensitivity of the call. Most work runs fine in ordinary or private mode; reach for `:tee` when you want a verifiable sealed execution, and `:e2e-tee` when the content must be encrypted to the sealed environment and you can accept its constraints.
