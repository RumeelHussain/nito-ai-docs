---
title: "What Nito Keeps"
description: "Nito's baseline retention posture. Only operational metadata is kept; request and response bodies are request-local and served with no-store."
---

Nito's default posture is **metadata only**: the gateway keeps operational metadata about a request but does not store the request or response body. This is the single most important retention fact about Nito — every stronger privacy mode builds on this floor rather than replacing it.

If you read only one sentence: **prompts, completions, reasoning, tool calls, and uploaded file contents are request-local and are not written to durable storage by default; what the gateway keeps is metadata.**

## What "Metadata" Means

It is worth being precise, because "metadata" is a word that can quietly expand to mean anything. On Nito, in the retention context, **metadata** is the operational record of _that a request happened and how it was handled_, not _what the request said_. It is the difference between knowing a call was made (when, which model, how many tokens, whether it succeeded, what it cost) and knowing the words in the prompt.

Concretely, the metadata-only posture means the gateway can keep operational facts such as:

- which model and provider served the call,
- token counts and usage for billing,
- timing, status, and error class,
- the request ID and similar correlation identifiers.

And it means the gateway does **not** persist the contents:

- prompt and message bodies,
- the model's completion,
- reasoning / chain-of-thought,
- tool calls and their arguments,
- the contents of uploaded files (parsed inline to serve the request, then discarded).

Those content categories are **request-local**: they exist for the lifetime of the request and are not saved to a durable store you could read back later. The full who-sees-what breakdown is on [Data visibility](/privacy/data-visibility).

## Three Enforcement Points

The metadata-only default is not just a policy statement; it shows up in code at concrete points.

<Steps>
  <Step title="Bodies are request-local" icon="1" iconType="solid" titleSize="h4">
    The request and response payloads are handled in memory to serve the call and are not committed to durable storage on the default path. There is no server-side prompt or completion store the gateway writes to on every request.
  </Step>
  <Step title="Responses carry no-store" icon="2" iconType="solid" stepNumber={2} titleSize="h4">
    Every inference response carries `Cache-Control: no-store` — an explicit instruction not to cache the body, by the gateway or any intermediary. Streaming responses additionally carry `no-cache` and disable proxy buffering.
  </Step>
  <Step title="Telemetry uses an allowlist" icon="3" iconType="solid" stepNumber={3} titleSize="h4">
    Observability is deliberately narrow: traces and telemetry attributes are restricted to a `SafeAttrs` allowlist, so only operational fields are recorded. Prompt and completion contents are not allowlisted, so they never leak into traces — the answer to the common worry that "logging" quietly captures everything.
  </Step>
</Steps>

## Does Nito Keep User Prompts?

Not by default. The metadata-only posture is grounded in observable code behavior — request-local bodies, the `no-store` header, and the telemetry allowlist — not a policy you have to take on faith. So when someone asks that blunt question, the answer is plain: your prompts and completions are not persisted by the gateway.

## What This Does and Does Not Cover

The metadata-only default is the **gateway's** posture. It is not a statement about what an upstream provider does with the call; that is governed separately by provider-side enforcement (`Store: false`, ZDR, or TEE) and is covered in [Data visibility](/privacy/data-visibility) and [Privacy overview](/privacy/overview). Nor does it cover client-side chat history, which lives in the browser (see [Data visibility](/privacy/data-visibility)).
