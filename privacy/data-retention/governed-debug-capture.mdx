Sometimes you need to debug a request, and debugging is easier when you can see what actually went through. That capability is in tension with a metadata-only default, which deliberately does **not** keep request bodies. **Governed debug capture** is the name for the controlled way to resolve that tension: an explicit, opt-in capture of request detail, kept only for a bounded time. This page describes the concept as designed, and is explicit that the mechanism and its duration are not confirmed in code.

## The concept, as designed

The privacy and retention standard describes the intended shape: debug captures are **explicit opt-in** and carry **bounded retention**. The design intent is the important part:

- **Opt-in, not automatic.** A debug capture is something you turn on deliberately for a request or a session. It is not the default, and the metadata-only posture remains the baseline. The default still does not persist prompts or completions; a debug capture is a conscious exception to that, made by someone with the authority to make it.
- **Bounded retention.** When a capture is taken, it is meant to live only for a limited window and then be removed, so that turning on debugging does not quietly create a permanent store of request bodies.
- **Governed.** The "governed" in the name signals that the capability is controlled, not freely available, so that the exception to metadata-only cannot be triggered casually.

That is the model to communicate: debugging is possible, but it is an audited, time-bounded, opt-in exception rather than a silent always-on log.

## What is NOT confirmed

Here is where accuracy matters. The standard says "bounded retention" **without a number**. No debug-capture subsystem and no specific retention constant (such as a 7-day value) were found in the gateway code scanned for this documentation. The `7d`-style constants that do exist in the code are unrelated to debugging (they are API-key expiry and session/invite TTLs), so they must not be borrowed to fill in a debug-capture window.

Do not state "7 days" as fact, and do not describe a capture subsystem as if it is implemented and verifiable. Describe the concept as designed and opt-in, and mark the duration and the mechanism as pending SME confirmation.

## What you can rely on today

- The **default remains metadata-only**: request bodies are not persisted unless an explicit exception is invoked. See [Metadata-only default](/privacy/data-retention/metadata-only-default).
- Debug capture is **designed as opt-in with bounded retention**, per the privacy standard.
- The **mechanism and the exact retention window are unconfirmed** and must not be quoted as a number until an SME confirms them.
