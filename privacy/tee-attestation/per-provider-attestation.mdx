description: "How NEAR (domain-based) and Phala (appid-based) attestation differ in subject, default proof format, and requirements, and what they share."
---

Two providers back Nito's TEE modes, NEAR and Phala, and both produce attestation evidence through the same gateway pipeline. They differ in one structural way: how they identify the attestation **subject**. This page lays out the difference and the shared machinery, so you know what to expect in the [response](/privacy/tee-attestation/attestation-pipeline) for each.

## The Attestation Subject

When a TEE produces an attestation, the signed statement has to say _what_ it is vouching for. The **attestation subject** is that identity anchor: the name the proof is bound to, so you can confirm the evidence belongs to the workload you meant to run and not some other environment. Think of it as the name on a notarized document. The notary's stamp (the hardware signature) proves the document is genuine; the name on it tells you whose document it is. NEAR and Phala both stamp the document, but they write the name in different fields.

Why two providers at all? Each runs its own TEE infrastructure, so each has its own way of naming the workload it sealed. The gateway normalizes everything else, but it cannot erase this structural difference, so it surfaces it to you in the response. Knowing the difference means you can read an attestation from either provider without surprise.

## What Providers Share

Both providers expose attestation through the same endpoint on their side, `attestation/report`, and the gateway parses both responses with a **shared parser**. The result is that, from the gateway's perspective, the two providers feed into the same [seven-gate baseline](/privacy/tee-attestation/seven-gate-baseline) and the same response shape. The differences are confined to how the subject is named and whether it is required.

## NEAR: Domain-based Attestation

NEAR uses **domain-based** attestation.

| Aspect | NEAR |
| --- | --- |
| Subject sent as | `domain` |
| Default proof format | `near-attestation-report-v1` |
| Subject required? | Optional |

The subject for NEAR is a **domain**, and it is **optional**. A NEAR attestation request can be made without specifying a subject, and the default proof format is `near-attestation-report-v1`.

## Phala: Appid-based Attestation

Phala uses **appid-based** attestation.

| Aspect | Phala |
| --- | --- |
| Subject sent as | `appid` |
| Default proof format | `redpill-attestation-report-v1` |
| Subject required? | Required |

The subject for Phala is an **appid**, and it is **required**. A Phala attestation must carry a subject; there is no subject-optional path the way there is for NEAR. The default proof format is `redpill-attestation-report-v1`.

## Side by Side Comparison

|  | **NEAR** | **Phala** |
| --- | --- | --- |
| Subject key | `domain` | `appid` |
| Subject requirement | Optional | Required |
| Default proof format | `near-attestation-report-v1` | `redpill-attestation-report-v1` |
| Provider endpoint | `attestation/report` | `attestation/report` |
| Parser | Shared | Shared |

## How This Surfaces in the Response

In the [attestation response](/privacy/tee-attestation/attestation-pipeline), the `proof_format` field is what tells you which provider produced the evidence: it defaults to `near-attestation-report-v1` for NEAR and `redpill-attestation-report-v1` for Phala, unless the evidence specifies otherwise. The `attestation_subject` field (when present) carries the subject that was sent — a `domain` for NEAR, an `appid` for Phala.

This per-provider distinction does not change the verification contract you run as a client. Whichever provider served the call, the [seven-gate baseline](/privacy/tee-attestation/seven-gate-baseline) and the [verify-yourself path](/privacy/tee-attestation/verify-yourself-recipe) apply the same way. The subject naming is something to be aware of when you read the response, not a separate verification flow.

Which provider backs a given model is reported per model as `attestation_provider` on its `GET /v1/models` entry (`near` or `phala`). Because that mapping is discovered from the providers at runtime, read the field rather than a static list. Each model carries exactly one `attestation_provider`, so no model is backed by both NEAR and Phala.
