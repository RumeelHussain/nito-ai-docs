When you send a prompt to an AI model, it normally runs on a computer you do not control, owned and operated by some provider. Even if that provider promises not to look at your prompt or keep it, you are taking their word for it. Nothing stops an administrator, a compromised server, or the operating system itself from reading the data while the model processes it. For most requests that is an acceptable risk. For sensitive ones, it is not. A Trusted Execution Environment is the hardware that turns "trust us" into "check for yourself." This page explains what it is, in plain terms, before the rest of this section gets into the mechanics.

## The problem a TEE solves

Software protections like encryption are good at protecting data while it is stored on disk or sent over a network. The hard case is data **in use**: the moment a computer actually runs a computation, it has to decrypt the data into memory to work on it. At that instant, whoever controls the machine, the operating system, or the hardware can in principle read it. A model cannot generate a response without processing your prompt, so your prompt is exposed exactly when the work happens. This is the gap a TEE closes.

## What a TEE is

A **Trusted Execution Environment (TEE)** is a region of a computer, enforced by the processor itself, that runs code in isolation from everything else on the machine. Code and data inside the TEE run normally, but the host operating system, the machine's owner, other programs on the same hardware, and even someone with physical access cannot read what is inside or change it. The isolation is created and policed by the silicon, not by software that the host could switch off. Think of it as a locked box welded shut at the factory: the computation happens inside, and even the person holding the box cannot open it or see in.

That isolation is valuable on its own, but by itself it would still just be a stronger promise. You would have to believe that the provider really put your computation inside a genuine sealed box and did not quietly run it somewhere ordinary. The second property of a TEE is what removes that leap of faith.

## What attestation adds

A TEE can produce an **attestation**: a signed statement, generated by the hardware, that describes what is running inside the sealed environment. Two things make it powerful:

- It is **signed by the hardware**, using a key rooted in the chip manufacturer, not by the provider's software. The provider cannot forge it.
- It commits to **what is running**, in the form of a measurement (a cryptographic fingerprint) of the exact code loaded into the environment.

So an attestation lets you confirm two facts you otherwise could not: that a real TEE produced the result, and that the code inside it was the code you expected. If the environment is not a genuine TEE, or the code inside has been swapped, the attestation will not check out. To continue the analogy: the locked box does not just keep its contents private, it also hands you a tamper-evident receipt, signed by the factory, proving it was that exact box that did the work.

## A freshness check, so the receipt is for your request

A signed receipt is only useful if it describes **your** computation and not an old one someone is replaying. To prevent that, you include a fresh, unpredictable value called a **nonce** with your request, and the TEE binds that nonce into the attestation. When you check the attestation, you confirm your nonce is present. If it is, the receipt was produced for this request, now, and not copied from an earlier one. You will see this nonce binding throughout this section; it is the reason every verification recipe starts by generating a new random value.

## What this means on Nito

Nito uses this pair, sealed execution plus a checkable attestation, to back its strongest privacy modes. When you select the `:tee` or `:e2e-tee` mode on a supported model, your call runs inside one of these environments, and you can fetch the attestation and verify it yourself rather than trusting a claim. That is the difference between a privacy promise and a privacy property you can confirm.

One honest caution, which the rest of this section expands on. Producing an attestation and **fully** verifying its hardware quote are two different jobs. The Nito gateway runs a set of baseline checks (the evidence is well-formed, the signing material is present, your nonce is bound) and returns the raw evidence to you. The deepest hardware checks, validating the manufacturer's quote and certificate chains, are yours to perform on that evidence if your threat model demands them. This is why these docs say a result is **attested** and **verifiable**, never "trustless." The assurance is real, and it is checkable, which is exactly the point.
