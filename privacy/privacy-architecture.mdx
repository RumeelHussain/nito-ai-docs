The [Privacy overview](/privacy/overview) makes the promises. This page shows how they are kept. It follows a single request from the moment it arrives at the gateway to the moment it is billed, and it points at the exact hop where each privacy property is enforced. The theme throughout: privacy is not a single switch flipped in one place; it is a sequence of constraints applied at every hop, so that a request cannot reach a provider in a weaker mode than you asked for.

## Hop 1: Authentication

Every request begins with authentication. You send your API key as a bearer token in the `Authorization` header; the gateway resolves it to your organization (the account that owns billing and keys) or rejects the request. From a privacy standpoint the relevant fact is what this hop establishes: an identity to bill and rate-limit against. The credential itself is never stored in raw form (tokens are stored as HMAC-SHA256 hashes), and the request body, your prompt, is not touched yet. Nothing about the prompt's contents influences authentication.

## Hop 2: Model and privacy-mode resolution

This is where the privacy mode is decided, and it is decided **once**. The gateway parses the `model` string. The base part selects the model; an optional `:confidential` or `:encrypted` suffix selects a stronger mode. A model sent with no suffix resolves to ordinary or private according to that model's stored tier. Any suffix other than `:confidential` or `:encrypted` is rejected here, before routing.

The output of this hop is a fixed privacy mode for the rest of the request's life. Nothing downstream can promote or demote it. If you asked for `:encrypted`, this is also where the gateway checks that the `X-Z-ENCRYPTED-*` encryption headers are present and that the request is non-streaming; if those constraints are not met, the request is rejected at this stage rather than served in a weaker form.

## Hop 3: Routing under the privacy filter

A single logical model can be served by more than one physical backend (the same model at different providers or at different privacy tiers). The router gathers the eligible backends and chooses one, but it filters **first by hard requirements**, and the privacy mode is a hard requirement. A `:confidential` request is filtered down to TEE-capable backends only; a `private`-tier request sets `RequireZeroDataRetention`, which in turn drives provider-side ZDR.

The load-bearing property here is the refusal to downgrade. If the privacy filter leaves zero eligible backends, the call **fails**. It does not fall back to a backend that cannot honor the mode. Only after the privacy filter is satisfied does the router order what remains by availability and cost. So routing optimizes for cost and health, but never at the expense of the privacy mode.

## Hop 4: The provider call, inside the chosen boundary

Once a backend is chosen, the gateway calls the provider inside the privacy boundary the mode requires, and each provider enforces privacy in its own way:

| Provider | Privacy enforcement at the boundary |
| --- | --- |
| xAI (Grok) | `Store: false` is forced on every request, so the upstream is told not to retain the exchange. |
| OpenRouter | A zero-data-retention data policy is applied when the route requires it. It is relaxed only when OpenRouter server-side web tools are used; embeddings and images force ZDR regardless. Models without a ZDR policy are never imported. |
| NEAR | Runs in a TEE; produces a domain-based attestation. Supports the end-to-end encrypted path. |
| Phala | Runs in a TEE; produces an appid-based attestation. Supports the end-to-end encrypted path. |
| FAL (images) | `X-Fal-Store-IO: 0` is sent so the upstream does not store image inputs or outputs. |

The point of the table is that the boundary is enforced **at the provider**, by a concrete mechanism, not merely asserted by the gateway. For TEE modes the boundary is hardware-isolated, and the provider emits an attestation that you can later verify.

## Where attestation fits

For `:confidential` and `:encrypted` calls, the sealed environment produces a signed attestation. You retrieve it through `GET /api/v1/attestation`, supplying at minimum the `model` and ideally a fresh random `nonce`. The gateway runs **baseline** verification on the evidence before returning it:

1. Provider evidence is present and parses as a JSON object.
2. A provider-declared `verified:false` marks the attestation unverified.
3. Proof format, signing algorithm, and model public key are present; if you passed `signing_algo`, it must match.
4. Your nonce matches the nonce the provider observed.
5. The gateway report-data binding holds: `report_data == signing_address + nonce`, with the requested address matching.
6. The nested model-attestation binding holds (model ID, algorithm, model signing identity, public key, nonce).
7. The nested JSON payload nonce for GPU envelopes is checked.

What this hop deliberately does **not** do is verify the full hardware quote: no Intel TDX quote validation, no NVIDIA GPU evidence checks, no certificate chains, CRLs, or TCB policy. Those are delegated to the client. The gateway returns the raw `evidence`, the signing address, the model public key, the proof format, and your echoed nonce, so you can run the deeper hardware verification independently. Use the words **baseline-verified** and **attested** for what the gateway does; the full assurance is the verify-yourself path. A reference verifier exists for exactly this purpose.

One nuance worth flagging for reviewers: anti-replay is **caller-driven**. The freshness binding relies on you supplying a fresh random nonce. If you omit the nonce, an empty-binding report is permitted, so the protection is only as strong as your nonce discipline.

## Hop 5: Metadata-only logging

Running alongside the request, not in its path, is logging and telemetry. The default posture is **metadata only**. Prompts, completions, reasoning, tool calls, and uploaded file contents are request-local and not persisted by the gateway. Telemetry is restricted to a safe allowlist of fields, and inference responses carry `Cache-Control: no-store`. This is the same floor described in [Zero data retention](/privacy/zero-data-retention); the architectural point is that the logging hop sees usage metadata, not content.

## Hop 6: Billing

Billing brackets the provider call. Before the call, the gateway places a hold against your credit balance sized to the expected cost. After the call, the hold is settled against actual usage, or released if the request failed before producing usage. Billing operates on usage metadata (token counts, request shape), not on the content of your prompt or the reply. So the most privacy-sensitive part of the request never has to be retained to be billed.

## The shape of the guarantee

Read top to bottom, the architecture is a funnel that only ever narrows: the mode is fixed at resolution, enforced as a hard filter at routing, realized by a concrete mechanism at the provider boundary, and kept out of logging and billing by the metadata-only posture. There is no hop at which a request becomes _less_ private than you asked for. The only thing that can happen if the mode cannot be honored is that the call fails.
